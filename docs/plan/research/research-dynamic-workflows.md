# Dynamic Workflows in Claude Code (from `.tmp/docs_claude/02-build-with-claude-code/01-agents-and-parallel-work/04-dynamic-workflows.md`)

## 1. What dynamic workflows are

A **dynamic workflow is a JavaScript script that orchestrates subagents at scale**. Claude writes the script for a task you describe; a **runtime executes it in the background in an isolated environment**, separate from the conversation — intermediate results live in script variables, not Claude's context. Claude's context only receives the final answer. Availability: **Claude Code v2.1.154+, all paid plans** (Anthropic API, Bedrock, Google Cloud Agent Platform, Microsoft Foundry); **on Pro it must be enabled via the "Dynamic workflows" row in `/config`**.

Positioning vs. alternatives (doc's comparison table): subagents/skills = Claude orchestrates turn-by-turn, results land in context, a few tasks per turn; agent teams = lead agent supervising peers via shared task list; **workflows = the script holds the loop/branching/intermediate results, dozens-to-hundreds of agents per run, resumable in the same session, and the orchestration itself is the repeatable artifact**. Key quality pattern: workflows can encode adversarial cross-verification (agents review each other's findings before reporting) or multi-angle plan drafting.

### Script anatomy
Saved file = `meta` block + plain-JS body with top-level `await`:
```javascript
export const meta = {
  name: 'audit-routes',
  description: 'Audit every route handler for missing auth checks',
}
const found = await agent('List every .ts file under src/routes/.', {
  schema: { type: 'object', required: ['files'], properties: { files: { type: 'array', items: { type: 'string' } } } },
})
const audits = await pipeline(found.files, file =>
  agent(`Audit ${file} for missing authentication checks.`, { label: file }),
)
return audits.filter(Boolean)
```
- **Primitives: `agent()` (spawn one subagent), `parallel()`, `pipeline(list, fn)` (one agent per item), `phase()`** — per the Agent SDK TypeScript reference (`.tmp/docs_claude/06-agent-sdk/08-sdk-references/01-typescript.md` §Workflow), an inline script "must begin with `export const meta = { name, description }` **as a literal**, followed by the script body using `agent()`, `parallel()`, `pipeline()`, and `phase()`. An optional `phases` array in `meta` groups agents under named stages in the progress view."
- `agent()` options shown in docs: `schema` (JSON Schema → structured output; changelog: schema-validation failures abort after 5 attempts) and `label`. The workflows doc defers "the full set of options" to the SDK reference's Workflow tool entry (not further enumerated in the local docs). The cost section implies per-stage model routing is possible ("Every agent in a workflow uses your session's model **unless the script routes a stage to a different one**").
- **Workflow tool (SDK v0.3.149+)**: input `{ script?, name?, scriptPath?, args?, resumeFromRunId? }` — at least one of script/name/scriptPath required; `name` resolves "a built-in workflow or one saved in `.claude/workflows/`"; `scriptPath` takes precedence and every invocation persists its script + returns the path (edit & re-invoke to iterate); `args` is exposed to the script as global `args` (real JSON values, `undefined` if omitted); `resumeFromRunId` replays cached `agent()` results for unchanged inputs, same session only. Output: `{ status: "async_launched", taskId, runId, summary, transcriptDir, scriptPath, error }` — **check `error`: a script failing its syntax check still returns `async_launched` but never runs**.

### Runtime behavior & limits (doc's constraint table)
- **No mid-run user input** — only agent permission prompts pause a run; stage sign-off = run each stage as its own workflow.
- **The script itself has no filesystem or shell access** — agents do all reads/writes/commands; the script only coordinates.
- **Max 16 concurrent agents** (fewer on limited CPU cores); **1,000 agents total per run**.
- Every run writes its script under `~/.claude/projects/<session>/`; Claude gets the path; you can diff/edit/relaunch.
- Resume works **within the same session only** (completed agents return cached results); exiting Claude Code means next session starts fresh.
- Determinism restriction (changelog evidence, v2.1.16x era): workflow **validation rejects scripts that call `Date.now()`/`Math.random()`** (a fix ensured mere mentions in strings/comments are no longer rejected — i.e., actual nondeterministic calls are validated against).
- Unverified-claim handling (v2.1.196+): claims verifier agents can't check (rate limit/API error) are reported as unverified, not refuted.

### Triggering
- **Prompt keyword `ultracode`** (before v2.1.160 the keyword was `workflow`) or **natural language** ("use a workflow", "run a workflow") — a direct request is treated as the same opt-in. Keyword trigger is controlled by setting `workflowKeywordTriggerEnabled` (default true, v2.1.157+).
- **`/effort ultracode`** = `xhigh` reasoning + automatic workflow orchestration for every substantive task; **session-only, resets on new session, NOT readable from settings.json** (set via `/effort`, `--settings`, or SDK control request); only on models supporting `xhigh`.
- Running a **bundled** (`/deep-research`, requires WebSearch tool) or **saved** workflow command.
- Doc explicitly: "If you already have an orchestrator built another way, such as **a folder of subagent prompts or a skill that fans work out, you can point Claude at it and ask for a workflow that does the same thing**."

### Approval & permissions
CLI per-run prompt shows planned phases: Yes / **"Yes, and don't ask again for `<name>` in `<path>`"** (per-workflow per-project consent) / View raw script (Ctrl+G opens editor, Tab adjusts prompt) / No. By permission mode: default & acceptEdits → every run unless don't-ask-again recorded; auto → first launch only (consent recorded in user settings; **skipped entirely when ultracode is on**); bypassPermissions / `claude -p` / Agent SDK → never prompted. **Workflow-spawned subagents always run in `acceptEdits` mode regardless of session mode, inherit your tool allowlist, and file edits are auto-approved**; non-allowlisted shell/webfetch/MCP calls still prompt mid-run → doc advises pre-adding needed commands to the allowlist before long runs.

### Saving / named workflows
`/workflows` → select run → `s`. Two locations: **`.claude/workflows/` in the project (shared with everyone who clones the repo)** or `~/.claude/workflows/` (personal, all projects). Saved workflows **run as `/<name>`** and appear in `/` autocomplete alongside bundled ones. Project beats personal on name collision. Monorepo (v2.1.178+): saves target the closest existing `.claude/workflows/` between cwd and repo root; **workflows load from every `.claude/workflows/` along that path**, closest wins. Saved workflows accept input via `args` ("Run /triage-issues on issues 1024, 1025, and 1030" — Claude passes structured data). The `.claude`-directory reference (`01-getting-started/04-core-concepts/03-claude-directory.md`) lists `workflows/*.js` as a first-class project-or-global config type: "Dynamic workflow scripts written by Claude and saved from /workflows; **each file becomes a `/<name>` command**."

### Disabling
`/config` toggle; `"disableWorkflows": true` in settings.json; `CLAUDE_CODE_DISABLE_WORKFLOWS=1`; org-wide via managed settings/admin page. When disabled: bundled workflow commands unavailable, `ultracode` keyword inert, ultracode removed from `/effort`. Also `disableBundledSkills`/`CLAUDE_CODE_DISABLE_BUNDLED_SKILLS` hides bundled skills, workflows, and built-in slash commands from the model. Works on CLI, Desktop, IDE extensions, `claude -p`, Agent SDK.

## 2. How a generated harness can leverage workflows

- **A harness CAN ship `.claude/workflows/*.js` files** checked into the repo: each becomes a `/<name>` slash command for every user who clones it. The file format is documented (meta block + JS body) and hand-editable, so a harness generator can author them directly rather than only saving from `/workflows`. Nothing in the docs requires workflows to be created via the save flow.
- **Plugins are NOT documented as able to ship workflows** — the plugins reference lists commands/agents/skills/hooks/MCP components but has zero workflow mentions; workflow load paths are only `.claude/workflows/` (project, walked up to repo root) and `~/.claude/workflows/`.
- **CLAUDE.md / skill instructions can trigger workflows in natural language** — "asking in your own words … works: Claude treats a direct request as the same opt-in" — so a skill can say "run a workflow that fans out one reviewer per changed file, then verify findings". A skill can also tell Claude to run a saved workflow by name with `args` (Claude uses the Workflow tool with `name` + `args`).
- Skills can adapt to effort via `${CLAUDE_EFFORT}` substitution (ultracode reports as `xhigh`, never a distinct level — same in hooks' `effort.level` and `$CLAUDE_EFFORT` env var).
- **End-user friction the harness must plan for**: per-run approval prompt in default/acceptEdits modes (mitigate: instruct users to pick "don't ask again", or document `permissions.allow` entries for the shell/MCP commands the workflow's agents need); paid-plan gating + Pro `/config` opt-in; user/org `disableWorkflows` → harness should degrade to plain subagent/skill orchestration.

## 3. Sibling docs: custom subagents & the rest of the directory

**`01-create-custom-subagents.md`** (the worker primitive workflows orchestrate):
- Locations/priority: managed settings (1) > `--agents` CLI JSON (2) > **`.claude/agents/` project (3)** > `~/.claude/agents/` user (4) > plugin `agents/` dir (5). Project agents are meant to be checked into version control. Directories scanned recursively (subfolders fine; identity comes from `name` frontmatter only; plugin subfolders become scoped ids like `my-plugin:review:security`). Nested `.claude/agents/` up to repo root all load; closest wins on collision (v2.1.178+).
- Format: Markdown + YAML frontmatter; body = the subagent's **system prompt** (replaces the Claude Code system prompt; subagent gets body + basic env details only). Required: `name`, `description`. Optional: `tools` (allowlist), `disallowedTools`, `model` (`sonnet|opus|haiku|fable|<full id>|inherit`, default inherit), `permissionMode`, `maxTurns`, **`skills` (preloads FULL skill content at startup)**, `mcpServers` (inline defs or references — inline keeps MCP tool descriptions out of main context), `hooks` (lifecycle hooks scoped to the subagent, e.g. PreToolUse validators that exit 2 to block), `memory` (`user|project|local` → persistent `agent-memory/<name>/` dirs; MEMORY.md first 200 lines/25KB injected), `background`, `effort`, `isolation: worktree`, `color`, `initialPrompt`.
- **Plugin subagents ignore `hooks`, `mcpServers`, `permissionMode`** (security) — a repo-level `.claude/agents/` harness keeps all fields, which favors shipping harnesses as repo `.claude/` trees over plugins when those fields matter.
- Skills↔subagents integration (both directions, same underlying system): subagent `skills:` field preloads skills; **skill `context: fork` + `agent: <type>`** runs the skill content as the prompt inside a named subagent (built-ins `Explore`/`Plan`/`general-purpose` or **any custom subagent from `.claude/agents/`**; default `general-purpose`).
- Invocation: automatic delegation via `description` (add "use proactively"); natural language; `@agent-<name>` mention (guarantees the subagent runs); `claude --agent <name>` or `"agent": "<name>"` in `.claude/settings.json` makes the whole session run as that agent. `Agent(type1,type2)` in `tools` restricts spawnable types (main-thread `--agent` only); `permissions.deny: ["Agent(name)"]` disables specific subagents; omitting `Agent` from tools prevents any spawning.
- Context: fresh context per spawn; loads CLAUDE.md hierarchy + git snapshot + preloaded skills — **except built-in Explore/Plan which skip CLAUDE.md and git status** (rules like "ignore vendor/" must be restated in the delegation prompt). Nested subagents (v2.1.172+) to depth 5. Background-by-default as of v2.1.198; permission prompts surface in the main session. Resumable via SendMessage/agent ID (Explore/Plan are one-shot, not resumable). `SubagentStart`/`SubagentStop` hooks in settings.json match on agent name. Live file-watch of agents dirs (restart needed only for a scope's first-ever agents dir, or with `--disable-slash-commands`).

**`00-overview.md`**: four parallelism surfaces — subagents (in-session delegation), agent view (`claude agents`, research preview, dispatch/monitor background sessions), agent teams (experimental, shared task list + messaging), dynamic workflows (script-driven, scale). Plus worktrees (file isolation; `isolation: worktree` for subagents) and `/batch` (bundled skill: 5–30 worktree-isolated subagents each opening a PR). `/workflows` is the check-in surface for workflow runs; `/tasks` for background items.

**`03-run-agent-teams.md`**: experimental, **disabled by default** (`CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` via env or settings.json `env`). Lead + teammates, shared task list (file-locked claiming), direct messaging. **Teammates can be spawned from any subagent definition scope** (tools+model honored, body appended to system prompt; `skills`/`mcpServers` frontmatter NOT applied to teammates). Team/task state under `~/.claude/teams/{session-name}/` and `~/.claude/tasks/` is runtime-generated — **cannot be pre-authored by a harness; no project-level team config exists** (a `.claude/teams/teams.json` is ignored). Quality-gate hooks: `TeammateIdle`, `TaskCreated`, `TaskCompleted` (exit 2 blocks/sends feedback). Teammates read CLAUDE.md normally. Limitations: no resumption of in-process teammates, one team per session, no nested teams, no background subagents from in-process teammates.

**`05-worktrees.md`**: `--worktree`/`EnterWorktree`; worktrees branch from `origin/HEAD` by default (`worktree.baseRef: "head"` to include unpushed work — relevant when isolating subagents operating on in-progress work); `.worktreeinclude` copies gitignored files (`.env` etc.) into new worktrees including subagent worktrees; `WorktreeCreate`/`WorktreeRemove` hooks replace git logic for other VCS.

**`06-ultraplan.md` / `07-ultrareview.md`**: cloud features (research preview) — `/ultraplan` drafts plans on Claude Code on the web; `/code-review ultra` runs a verified multi-agent review in a cloud sandbox ($5–20/run after 3 free runs, never auto-triggered by Claude). Both unavailable on Bedrock/GCP/Foundry. Mostly out of scope for local harnesses but `claude ultrareview` has a non-interactive CI subcommand.

## 4. Constraints & gotchas checklist for generated harnesses

1. **Workflows are opt-in per task**: at normal effort Claude won't write a workflow unless the prompt asks (keyword `ultracode` or natural language) or a saved/bundled workflow command is invoked. `ultracode` effort cannot be persisted in settings.json (session-only).
2. **Caps**: 16 concurrent agents (CPU-dependent), 1,000 agents/run, no mid-run user input, script has no direct fs/shell, resume same-session only, scripts syntax-checked at launch (`error` field despite `async_launched`), nondeterministic calls (`Date.now()`/`Math.random()`) rejected by validation, `agent({schema})` retries capped at 5.
3. **Permissions**: workflow subagents always run `acceptEdits` + inherit allowlist → a harness-shipped workflow will auto-edit files once approved; pre-populate `permissions.allow` for Bash/WebFetch/MCP calls the agents need or the run stalls on prompts.
4. **Availability gating**: v2.1.154+ / paid plans / Pro `/config` opt-in / can be disabled by user (`disableWorkflows`, env var) or org (managed settings) → design fallbacks.
5. **Version sensitivity**: keyword was `workflow` pre-2.1.160; monorepo `.claude/workflows/` walking is 2.1.178+; Workflow tool needs SDK 0.3.149+; `/agents` wizard removed in 2.1.198 (edit files directly); subagent background-default 2.1.198.
6. **Plugin limits**: plugin agents lose `hooks`/`mcpServers`/`permissionMode`; plugins can't ship workflows (undocumented) → prefer repo-level `.claude/` for full-fidelity harnesses.
7. **Explore/Plan skip CLAUDE.md** → harness rules do not reach them; restate critical rules in delegation prompts or use custom agents.
8. Glossary note: a **verification loop** ("a check Claude can run") is called the prerequisite for `/goal`, unattended runs, and dynamic workflows — harness workflows should embed verifiable checks, mirroring the fan-out→adversarial-verify→synthesize pattern of `/deep-research`.

## KEY FACTS
- A dynamic workflow is a JavaScript script (written by Claude, executed by an isolated background runtime) that orchestrates subagents at scale; intermediate results stay in script variables, only the final result reaches Claude's context. Requires Claude Code v2.1.154+, paid plans; Pro users must enable it in /config.
- Workflow script format: file starts with a literal `export const meta = { name, description }` (optional `phases` array groups agents into named stages), followed by plain JS with top-level await using the primitives agent() (one subagent; options include `schema` for JSON-Schema structured output and `label`), pipeline(list, fn) (one agent per item), parallel(), and phase().
- Saved workflows live in `.claude/workflows/*.js` (project scope, shared with everyone who clones the repo) or `~/.claude/workflows/` (personal); each file becomes a `/<name>` slash command in autocomplete; project beats personal on name collision; in monorepos (v2.1.178+) every `.claude/workflows/` from cwd up to repo root loads and the closest wins — so a harness can ship workflows by checking .js files into `.claude/workflows/`.
- Saved workflows accept invocation-time input via `args` (exposed to the script as a global; structured JSON, `undefined` if omitted), e.g. 'Run /triage-issues on issues 1024, 1025, and 1030'.
- The Workflow tool (Agent SDK v0.3.149+) takes { script | name | scriptPath, args, resumeFromRunId }; `name` resolves built-in or `.claude/workflows/` workflows; output is { status: 'async_launched', taskId, runId, transcriptDir, scriptPath, error } — a failed syntax check still returns async_launched but with `error` set and never runs; resumeFromRunId caches completed agent() calls (same session only).
- Triggering workflows is opt-in: prompt keyword `ultracode` (was `workflow` before v2.1.160) or natural-language requests like 'use a workflow'; `/effort ultracode` (xhigh + auto workflow orchestration) is session-only and cannot be set in settings.json; bundled `/deep-research` requires the WebSearch tool.
- Runtime limits: max 16 concurrent agents (fewer on limited CPUs), 1,000 agents total per run, no mid-run user input (run each stage as its own workflow for sign-off), the script itself has no filesystem/shell access (agents do the work), resume works only within the same Claude Code session, and validation rejects scripts calling Date.now()/Math.random() (determinism restriction, per changelog).
- Workflow-spawned subagents ALWAYS run in acceptEdits mode and inherit the session tool allowlist regardless of the user's permission mode; file edits are auto-approved; non-allowlisted Bash/WebFetch/MCP calls still prompt mid-run — harnesses should pre-add needed commands to permissions.allow.
- Launch approval by permission mode: default/acceptEdits prompt every run unless the user picked 'Yes, and don't ask again for <name> in <path>'; auto mode prompts first launch only (skipped entirely under ultracode); bypassPermissions, claude -p, and Agent SDK never prompt.
- Workflows can be disabled via /config toggle, `"disableWorkflows": true` in settings.json, CLAUDE_CODE_DISABLE_WORKFLOWS=1, or org managed settings — which removes bundled workflow commands, deactivates the ultracode keyword, and removes ultracode from /effort; harnesses need a fallback path.
- Custom subagents are Markdown files with YAML frontmatter in `.claude/agents/` (project, priority 3), `~/.claude/agents/` (user, priority 4), plugin agents/ dirs (priority 5), --agents CLI JSON (priority 2), or managed settings (priority 1); only `name` and `description` are required; the body is the subagent's system prompt; key optional fields: tools, disallowedTools, model, permissionMode, maxTurns, skills (preloads FULL skill content), mcpServers, hooks, memory (user/project/local persistent agent-memory dirs), background, effort, isolation: worktree, color, initialPrompt.
- Plugin-distributed subagents ignore the hooks, mcpServers, and permissionMode frontmatter fields for security — repo-level `.claude/agents/` keeps all fields, and plugins are NOT documented as able to ship workflows at all, so full-fidelity harnesses are better shipped as a repo `.claude/` tree than as a plugin.
- Skills and subagents compose in both directions: a subagent's `skills:` field injects full skill content at startup, and a skill with `context: fork` + `agent: <type>` runs the skill body as the prompt inside any built-in or custom subagent from `.claude/agents/` (default general-purpose).
- Built-in Explore and Plan subagents skip CLAUDE.md and git status entirely (not configurable) — harness rules in CLAUDE.md do not reach them and must be restated in delegation prompts or replaced by overriding a subagent named Explore.
- Subagents can be invoked by automatic delegation (description-driven; 'use proactively' encourages it), natural language, @agent-<name> mention (guaranteed), or `claude --agent <name>` / the `agent` setting in .claude/settings.json (whole session runs as that agent, its prompt replacing the default system prompt); permissions.deny 'Agent(name)' disables specific subagents; Agent(type1,type2) in tools restricts spawnable types.
- Agent teams are experimental and disabled by default (CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1); teammates can reuse subagent definitions (tools+model honored, body appended; skills/mcpServers not applied); team/task state under ~/.claude/teams/ and ~/.claude/tasks/ is runtime-generated and cannot be pre-authored by a harness; quality-gate hooks TeammateIdle/TaskCreated/TaskCompleted exit-2 to block.
- The doc explicitly endorses converting existing orchestrators: 'If you already have an orchestrator built another way, such as a folder of subagent prompts or a skill that fans work out, you can point Claude at it and ask for a workflow that does the same thing' — natural-language workflow requests from skill/CLAUDE.md instructions count as the same opt-in as the ultracode keyword.
- The workflowKeywordTriggerEnabled setting (default true, v2.1.157+) controls only the ultracode prompt keyword; saved workflow commands and /workflows are unaffected; disableBundledSkills / CLAUDE_CODE_DISABLE_BUNDLED_SKILLS hides bundled skills, workflows, and built-in slash commands from the model.
- In hooks, Bash subprocesses, and skill string substitution, ultracode is not a distinct effort level and reports as xhigh (CLAUDE_EFFORT / ${CLAUDE_EFFORT} / hook effort.level).
- The glossary names a verification loop (a check Claude can run, e.g. tests/build) as the prerequisite for /goal, unattended runs, and dynamic workflows — generated harness workflows should embed a runnable check and/or the fan-out→adversarial-verify→synthesize pattern used by /deep-research.

## OPEN QUESTIONS
- Whether agent() inside a workflow script can reference a named custom subagent type from .claude/agents/ or set per-call model/tools — the local docs only show `schema` and `label` options and defer 'the full set of options' to the SDK Workflow tool entry, which doesn't enumerate them either (the cost section implies per-stage model routing exists).
- Exact signatures/semantics of parallel() and phase() are not documented anywhere in the local docs corpus (only named as available primitives).
- Whether the per-workflow 'don't ask again for <name> in <path>' consent can be pre-seeded by a harness (its storage location — user settings for auto mode — is mentioned but the key/format is undocumented), i.e., can a harness ship zero-prompt workflow launches in default mode?
- Whether plugins/marketplaces can distribute workflow scripts at all — the plugins reference lists no workflows/ component, so distribution appears limited to repo .claude/workflows/ and ~/.claude/workflows/; unconfirmed whether this is a deliberate gap or upcoming.
- The full determinism rule set for workflow validation (Date.now/Math.random rejection is only known from a changelog bugfix line; no doc page states the complete list of banned APIs).
- Whether a skill's frontmatter (e.g., allowed-tools including Workflow) or a hook can programmatically launch a workflow without user approval in default permission mode — docs only describe launch via user prompt, saved/bundled commands, ultracode, or headless/SDK contexts.
