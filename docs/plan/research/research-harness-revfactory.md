## What harness_revfactory IS

**Identity.** `revfactory/harness` ("Harness — The Team-Architecture Factory for Claude Code"), a Claude Code **plugin** whose entire product is one meta-skill: `skills/harness/SKILL.md` (458 lines, Korean) + 6 reference docs. Self-positioned as "L3 Meta-Factory / Team-Architecture Factory" (a self-invented ecosystem taxonomy, `README.md` §Category). Apache 2.0, v1.2.0, author robin/Hwang (Kakao). Trilingual READMEs (EN/KO/JA), landing page `index.html`, marketing-heavy.

**Concept & end-to-end flow.** User installs via `/plugin marketplace add revfactory/harness`, exports `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1`, then says "build a harness for this project" (or Korean/Japanese equivalents). The meta-skill's description triggers, and Claude executes an 8-phase instruction workflow (`skills/harness/SKILL.md`):
- **Phase 0 (audit):** read existing `.claude/agents/`, `.claude/skills/`, `CLAUDE.md`; branch into new-build / extend / operate-maintain; detect drift between CLAUDE.md records and actual files; report to user and **get plan confirmation**.
- **Phase 1 (domain analysis):** parse domain from user request, explore codebase, detect conflicts with existing agents/skills, and **detect user proficiency** from conversational cues to calibrate jargon.
- **Phase 2 (architecture):** choose execution mode — **Agent Teams is the hard default** (TeamCreate/SendMessage/TaskCreate), subagents the fallback, hybrid per-phase mixing allowed — then choose one of **6 patterns**: Pipeline, Fan-out/Fan-in, Expert Pool, Producer-Reviewer, Supervisor, Hierarchical Delegation (`references/agent-design-patterns.md` has per-pattern team-mode suitability notes and a decision tree).
- **Phase 3 (agents):** after a **duplicate-check against existing agents (3-0)** with a reuse decision table, write every agent to `.claude/agents/{name}.md` — mandatory even for builtin types; mandatory `model: "opus"`; required sections: core role, working principles, I/O protocol, error handling, collaboration, plus a Team Communication Protocol section in team mode.
- **Phase 4 (skills):** after **duplicate-check (4-0)**, generate `.claude/skills/{name}/SKILL.md` per Anthropic-style rules: "pushy" descriptions (only trigger mechanism; boundary conditions; the PDF good/bad example), why-not-ALWAYS/NEVER, <500-line bodies, Progressive Disclosure into `references/`, script bundling when repeated helpers observed.
- **Phase 5 (orchestration):** one orchestrator skill from 3 templates in `references/orchestrator-template.md` (Team A / Subagent B / Hybrid C). Includes data-transfer protocol table (message/task/file/return-value), `_workspace/` file convention `{phase}_{agent}_{artifact}.md` with workspace preserved for audit, error handling (retry once → proceed with omission noted; conflicting data kept with sources), team-size guide (2-3/3-5/5-7; "3 focused teammates beat 5 scattered ones"), **CLAUDE.md pointer registration** (trigger rule + change-history table ONLY — explicitly not agent/skill lists), and **follow-up support**: orchestrator description must include re-run/update/fix keywords, and orchestrator Phase 0 must check `_workspace/` existence to route initial / partial-rerun / new-run (archive old workspace).
- **Phase 6 (validation):** structure checks, per-mode checks, execution tests with **with-skill vs without-skill A/B**, **trigger validation with 8-10 should-trigger + 8-10 near-miss should-NOT-trigger queries**, dry-run, and a mandatory Test Scenarios section (normal + error).
- **Phase 7 (evolution):** post-run feedback offer (never forced), a **feedback-type → modification-target routing table** (output quality→skill; role gap→agent; ordering→orchestrator; trigger miss→description), change-history logging in CLAUDE.md, proactive evolution triggers (same feedback 2x, repeated failures, user bypassing orchestrator), and a maintenance workflow (audit → one-change-at-a-time → sync → verify).
- Ends with an 18-item output checklist.

**References (progressive disclosure of the meta-skill itself, all under `skills/harness/references/`):**
- `agent-design-patterns.md` — modes, 6 patterns + composites, agent-type selection, agent split criteria (4 axes), agent-reuse table, skill-vs-agent distinction ("skill = how, agent = who"), 3 skill↔agent connection methods (Skill tool / inline / reference-load).
- `orchestrator-template.md` — full markdown templates A/B/C with hybrid transition rules (must TeamDelete before subagent calls; one active team per session).
- `skill-writing-guide.md` — description patterns, why-first, generalization-vs-overfitting, output-format/example patterns, PD patterns (domain-split references, conditional detail, ToC for >300-line refs), script-bundling signals (same helper 3/3 runs → bundle), **eval data schemas** (`eval_metadata.json`, `grading.json` with text/passed/evidence fields, `timing.json`), "what NOT to put in skills" (no READMEs, no meta-info, nothing Claude already knows), skill-reuse/generalization boundaries ("stop at intended responsibility scope").
- `skill-testing-guide.md` — with/without-skill parallel spawning, baseline selection, assertion authoring (discard non-discriminating assertions), Grader/Comparator(blind A/B)/Analyzer helper agents, iteration loop with immutable `iteration-N/` dirs, **description auto-optimization with 60/40 train/test split, max 5 iterations, select on test set to prevent overfit**.
- `qa-agent-guide.md` — the standout: **boundary-mismatch taxonomy** (API response shape vs frontend hook generic type; camelCase vs snake_case; file path vs href; state-transition map vs actual `.update({status})` calls; endpoint↔hook 1:1 mapping; 202-immediate vs async-final shape), why static review/`npm run build` misses these, "**read both sides simultaneously**" rule, use general-purpose not Explore for QA, **incremental QA after each module** not end-of-build, a checklist template ("existence checks are weak, cross-comparisons are strong"), a QA agent definition template — all sourced from **7 named real bugs** in the author's SatangSlide project.
- `team-examples.md` — 5 worked examples (research fan-out team, SF-novel pipeline with team re-formation between phases, webtoon producer-reviewer as subagents with max-2-retry / force-PASS rule, code-review team with peer-to-peer cross-domain messaging, supervisor migration team with task-claiming).

## harness-100/ (embedded companion catalog)

`harness-100/{en,ko}/` = 100 harnesses × 2 languages (1,808 md files; claims: 978 agent defs, 630 skills), 10 categories (content, dev/devops, data/AI, business, education, legal, lifestyle, docs, ops, specialized). Every harness: `NN-name/.claude/{CLAUDE.md, agents/*.md (always 4-5), skills/{orchestrator,domain-skill-1,domain-skill-2}/skill.md}`. Three-layer skill system: **orchestrator skill** (team coordination) + **agent-extending skills** (domain knowledge, e.g. `rca-methodology`, `route-optimizer`) + external tools. Verified examples: `en/25-incident-postmortem/` (5 agents, dependency DAG with parallel 2a/2b, **scale modes** mapping request patterns to agent subsets — full/timeline/RCA/remediation/review — plus reuse-existing-file skipping, blameless-culture enforcement rule) and `en/76-travel-planner/` (4 agents; `route-optimizer` skill is a ~130-line knowledge dump of time-block templates and city transport tips). Agents are ~100-140 lines with inline deliverable templates and team-communication protocols.

## How generation actually works

No questionnaire, no scoring rubric, no template files copied verbatim. It is **pure instruction-following**: markdown templates embedded in references + decision tables (mode tree, phase-selection matrix for extensions, reuse tables, team-size table). The "+60% quality / 15/15 wins / -32% variance" evidence is an author-measured n=15 LLM-judged A/B in a sister repo (`revfactory/claude-code-harness`); to its credit the README pairs every citation with the "n=15, author-measured, third-party replications pending" disclosure.

## Critical assessment

**Genuinely good, worth borrowing:**
1. **Phase 0 audit + drift detection + new/extend/maintain branching** — treats the generator as re-entrant, not one-shot. The dedup gates (3-0/4-0) with reuse decision tables directly attack the real failure mode of meta-generators: duplicate agents/skills accumulating under different names.
2. **CLAUDE.md pointer policy** — register only trigger rule + change history; single-source-of-truth for lists lives in the filesystem/orchestrator. They learned this the hard way (v1.1.0 registered full context; v1.2.0 reverted to pointers — see CHANGELOG.md).
3. **Follow-up-run design** — orchestrator descriptions must contain re-run/update keywords ("otherwise the harness is dead code after first run") and orchestrator Phase 0 routes initial/partial-rerun/new-run off `_workspace/` state. This is a sharp, non-obvious insight about skill lifecycle.
4. **Trigger-eval doctrine** — near-miss should-NOT-trigger queries as the only valuable negative tests; train/test split for description optimization to prevent overfitting.
5. **qa-agent-guide.md** — cross-boundary comparison checklists grounded in 7 real bugs; "read both sides simultaneously"; incremental QA. Best content in the repo.
6. **Eval hygiene** — with/without-skill A/B, assertion schemas, pruning non-discriminating assertions, bundling repeated helper scripts, immutable iteration dirs, blind Comparator agent.
7. **Feedback→target routing table + change-history log** as the evolution mechanism; evolution triggers including "user observed bypassing the orchestrator".
8. **_workspace/ conventions** — numbered `{phase}_{agent}_{artifact}` files, preserved for audit; conflicts kept with sources cited, never silently resolved.
9. **Scale modes** in generated orchestrators (map request patterns to agent subsets; skip agents when user supplies existing files) — good cost/latency pragmatism.
10. The **CHANGELOG itself** is a catalog of meta-generator lessons learned, useful as design input.

**Gimmicky / overbuilt / fragile:**
1. **Vaporware & dead links:** `/harness:evolve` advertised in KO/JA READMEs with a diagram but doesn't exist; 60 harness-100 skills point at nonexistent `references/*.md`; `docs/cost-controls.md` "forthcoming"; `[P-13](#)` placeholders; SLA-enforcement GitHub Action doesn't exist.
2. **harness-100 is cookie-cutter volume theater:** every one of 200 harnesses is exactly 4-5 agents + 3-4 skills with identical skeletons; zero bundled scripts/references/assets anywhere, contradicting the factory's own Progressive Disclosure mandate; the '1,808 files' number is the point. Domain skills are name-dropped frameworks (Porter, SM-2, 5 Whys, city transit tips) — knowledge Claude already has, violating their own "don't include what Claude knows" rule and burning context.
3. **skill.md vs SKILL.md case bug** across all 630 generated skills — works on macOS, likely breaks on Linux; the factory fails its own output checklist.
4. **Experimental-API bet + maximalism:** Agent Teams as default for everything, `model: "opus"` mandated everywhere, 5-agent teams for travel planning — enormous token cost for marginal benefit; their own guidance ("3 focused > 5 scattered") contradicts their own catalog (always 5).
5. **Process bloat:** 8 phases, ~20 sub-phases, an 18-item checklist, all executed purely by LLM instruction-following with no enforcement tooling — harness-100's violations prove the checklist isn't reliably followed even by its own author's runs.
6. **Marketing/engineering blur:** repo doubles as a growth-hacking operation (launch plans with star KPIs, HN drafts, awesome-list submission maps in _workspace/); the headline metric is an author-measured n=15 self-experiment; 'L3 Meta-Factory' is self-coined category framing; SLA tables are promises without machinery.
7. **Korean-only source of truth** for the meta-skill while marketing is trilingual — fine for Claude, but a maintainability/contribution barrier.

**Anti-patterns to avoid in our own harness-creator:** advertising unshipped features; emitting pointers to files you didn't generate; fixed-shape teams regardless of domain; multi-agent-by-default; regenerating encyclopedia knowledge into 'skills' instead of encoding project-specific contracts/decisions; unenforced checklists as quality theater; parallel agents writing shared files without ownership boundaries (their own post-M0 audit caught exactly this conflict).

**Elicitation (question 5):** No AskUserQuestion or structured intake. Interaction points are: plan confirmation after Phase 0 audit; inferred **user-proficiency tone calibration** (Phase 1-5 — a genuinely nice idea); user review as the qualitative eval channel in Phase 6-3 with an iterate-until-satisfied loop; a mandatory-but-optional post-run feedback ask in Phase 7-1 ("offer, never force; empty = fine"); and generated harnesses handle missing inputs via error-handling rows ('ask additional questions, tag [Unconfirmed]', 'propose 3 destinations/directions and let the user choose'). Up-front requirements gathering from the user is essentially absent — the domain sentence + codebase exploration is the whole intake. That is the clearest differentiation opportunity for a competing harness generator.

## OPEN QUESTIONS
- Does our harness-creator want to depend on the experimental Agent Teams API at all, or generate single-agent + skills harnesses that run on GA Claude Code? (revfactory's design collapses without CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1 — their own enterprise FAQ recommends 'design-time only' use as the workaround.)
- Should our generator perform structured up-front elicitation (AskUserQuestion) before generating? revfactory has none — it infers everything from one sentence plus codebase exploration, and defers missing-info handling to the generated agents' error-handling rows. This is the biggest obvious gap to differentiate on.
- How much of the 'validation' phase (with/without-skill A/B, trigger evals, assertion grading) do we actually want to execute vs merely instruct? revfactory specifies it in prose but ships no tooling, and harness-100 shows the checklist was not enforced on its own output.
