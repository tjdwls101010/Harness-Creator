## 2. Design philosophy in SKILL.md

The document is unusually self-aware — it is written to *demonstrate* the philosophy it teaches. Core tenets (all in `SKILL.md`):

- **Loop, not writing exercise** (§1): skill creation = decide → draft → run with/without skill on realistic prompts → evaluate qualitatively + quantitatively → rewrite → repeat. Claude's job is to locate the user in this loop and move them forward; the user can opt out ("skip the evals, just vibe") and the loop yields.
- **Conviction over compliance** (§2): principles (what + convincing why + concrete picture) beat rules because a persuaded model generalizes to unlisted cases; a rule is "a rail bolted on" that collides with model judgment on unforeseen cases. Litmus test: "given only the why, could the model re-derive the instruction and handle the case I forgot?"
- **The mechanism**: every instruction should carry *what*, *why*, and *what it looks like*. Includes a worked contrast (flat vs convincing ISO-8601 date rationale). Numbers must carry their reason and when to abandon them ("a number stripped of its reason is a rail wearing a digit").
- **Persuade only what isn't already believed**: don't re-teach what a competent model knows — over-explaining signals distrust and "clips intelligence from the other side." A skill holds only what the model can't supply: domain knowledge, unguessable preferences, decisions already made. Highest-signal content = **gotchas** (domain traps the model learns only by failing). Per-line keep/cut table; notably: cut "denials of things never raised."
- **Anti-brittleness corollaries**: don't enumerate 15 scenarios ("snaps on the sixteenth") — make 2-3 cases vivid, state the principle, stop. Length is not the enemy; non-contributing length is. Audit: what changes if the paragraph vanished?
- **Progressive disclosure has an optimum, not a direction** (§3): 3 load stages (metadata always / SKILL.md body on trigger / bundles on reach). Split by *invocation pattern*, never by volume; over-splitting creates routing failures ("a skill that can't reliably find its own knowledge fails silently"). Organize references by variant the model chooses between (`references/{aws,gcp,azure}.md`).
- **Scripts must compose**: parameterized CLIs / importable helpers, never frozen single-purpose scripts. Skills can carry state (`config.json` + NOT_CONFIGURED check idiom; append-log/SQLite memory under `${CLAUDE_PLUGIN_DATA}`).
- **One hard line**: no deception/harm; behavior must match the description the user agreed to.
- **Ask far more than feels natural** (§4): deliberately inverts Claude's don't-bother-the-user default inside skill creation — question costs seconds, wrong guess costs hours. Calibrate vocabulary to the user (non-coder to staff engineer).
- **Description = the sole trigger mechanism**; "when to use" info in the body is useless (loads too late). Claude currently *under*-triggers, so lean descriptions toward triggering.
- **Small eval sets first** (2-3 prompts you study deeply), scale later; assertions only for objectively checkable outputs — forcing assertions onto subjective ones "just measures noise."
- **Improvement doctrine** (§6): generalize from feedback (don't tighten rules, reach *wider*); read transcripts not just outputs; recover the *why* behind terse feedback; accumulate gotchas over time.

## 3. The full pipeline

### 3a. Two distinct eval systems

**System A — Output-quality eval loop (orchestrated by Claude in-session, no script drives it):**
1. Workspace: `<skill-name>-workspace/iteration-N/eval-<descriptive-name>/{with_skill,without_skill|old_skill}/run-*/`. For skill improvement, baseline = a snapshot (`cp -r` before editing).
2. Claude spawns **executor subagents in pairs (with-skill + baseline in the same turn** to cancel machine-load timing skew). Each saves outputs to `.../outputs/`, plus `transcript.md`, `metrics.json`, `user_notes.md`.
3. Claude saves `timing.json` per run **immediately from the subagent completion notification** (`total_tokens`, `duration_ms`) — the only moment those numbers exist.
4. Assertions drafted *while runs are in flight*, saved into `eval_metadata.json` per eval dir.
5. **Grader** (subagent following `agents/grader.md`): reads transcript + outputs, PASS/FAIL each expectation with evidence (burden of proof on the expectation; superficial compliance = FAIL), extracts and verifies implicit claims, reads `user_notes.md`, and critiques the evals themselves (`eval_feedback` suggestions). Writes `grading.json`. Viewer requires exact field names `text`/`passed`/`evidence`.
6. **`aggregate_benchmark.py <dir> --skill-name X [--skill-path P] [-o OUT]`**: discovers config dirs dynamically (any subdir containing `run-*`), pulls pass_rate/time/tokens/tool_calls/errors from each `grading.json` (falling back to sibling `timing.json` for time/tokens, `output_chars` as token proxy), computes mean/stddev/min/max per config, delta between first two configs (**order-dependent**: with_skill dir must sort before baseline — the SKILL.md says "put each with_skill config before its baseline"), writes `benchmark.json` + `benchmark.md`. Metadata models are left as `"<model-name>"` placeholders; `notes` left empty for the analyzer.
7. **Analyzer** (second half of `agents/analyzer.md`): reads benchmark.json, writes freeform notes (assertions that pass in both configs, high-variance evals, cost-vs-pass-rate tradeoffs) into `notes`.
8. **Viewer**: `nohup python eval-viewer/generate_review.py <workspace>/iteration-N --skill-name X --benchmark .../benchmark.json [--previous-workspace .../iteration-N-1] [--port 3117] [--static out.html] &`. Serves self-contained HTML; regenerates on each GET (refresh picks up new runs); kills prior listener on the port via lsof; falls back to ephemeral port. Feedback POSTs to `/api/feedback` → `feedback.json` (`status: in_progress` on auto-save, `complete` with ALL runs on submit so empty-vs-unreviewed is distinguishable). Static mode downloads feedback.json for headless use.
9. Claude reads `feedback.json`, applies changes, reruns as `iteration-N+1`. Stop when user happy / feedback empty / progress stalls.
10. **Optional blind comparison** (§8): `agents/comparator.md` judges A vs B without labels using a self-generated content+structure rubric (1-5 scales → overall 1-10, "be decisive, ties rare"), then `agents/analyzer.md` (first half) unblinds and produces prioritized `improvement_suggestions` with categories (instructions/tools/examples/error_handling/structure/references) and priorities keyed to "would it have changed the outcome."

**System B — Trigger/description optimization (fully scripted):**
1. Build ~20 trigger queries (even should/shouldn't split, effort on near-misses), user reviews via `assets/eval_review.html` → exports `eval_set.json`.
2. **`run_eval.py`** — flags: `--eval-set` (req), `--skill-path` (req), `--description` (override), `--num-workers` (10), `--timeout` (30s), `--runs-per-query` (3), `--trigger-threshold` (0.5), `--model`, `--verbose`. Mechanism: writes a uniquely-named command file (`{name}-skill-{uuid8}.md`) into `<project-root>/.claude/commands/` so it appears in available_skills; runs `claude -p <query> --output-format stream-json --verbose --include-partial-messages` with `CLAUDECODE` env var stripped (to allow nesting); parses the stream for `content_block_start` tool_use of `Skill` or `Read` and checks whether the accumulating `input_json_delta` contains the planted name (early exit); falls back to full assistant-message parsing; runs queries in a `ProcessPoolExecutor`; pass = trigger_rate on the right side of threshold; cleans up the command file in a finally.
3. **`run_loop.py`** — flags: `--eval-set`, `--skill-path` (req), `--model` (req), `--description`, `--num-workers` 10, `--timeout` 30, `--max-iterations` 5, `--runs-per-query` 3, `--trigger-threshold` 0.5, `--holdout` 0.4, `--verbose`, `--report auto|none|path`, `--results-dir`. Stratified train/test split (seed 42); each iteration evaluates train+test in one batch, records history, writes a live auto-refreshing HTML report (`generate_report.py`, opened in browser immediately), stops early if all train pass; **blinds test scores from the improver**; final best chosen by *test* score. Emits results JSON to stdout + optional timestamped results dir with `results.json`, `report.html`, `logs/improve_iter_N.json`.
4. **`improve_description.py`** — flags: `--eval-results`, `--skill-path`, `--history`, `--model` (req), `--verbose`; core function called by run_loop. Prompt includes: what a skill is, current description, failed-to-trigger and false-trigger queries with rates, full attempt history ("do NOT repeat — try something structurally different"), full SKILL.md body, anti-overfitting guidance (generalize to intent categories, not query lists), 100-200 word target, 1024-char hard limit, style tips (imperative, intent-over-implementation, distinctiveness). Calls `claude -p --output-format text` with prompt on **stdin**; parses `<new_description>` tags; one-shot shortening retry if >1024 chars; logs full transcripts per iteration.
5. **`generate_report.py`** — CLI: `input(json|-)  [-o out.html] [--skill-name]`; also imported by run_loop for live reports. Iteration×query matrix with train/test color coding, per-cell trigger-rate fractions, precision/recall in verbose stderr output (in run_loop).
6. Apply `best_description` to frontmatter; optionally **`package_skill.py <skill-folder> [output-dir]`** (positional args) → validated, filtered `.skill` zip.

**Support:** `quick_validate.py <dir>` (positional) — structural frontmatter validation, used standalone and by package_skill. `utils.parse_skill_md` is the shared frontmatter parser (hand-rolled, handles `|`/`>` multiline).

### 3b. Connection map
- SKILL.md orchestrates System A conversationally (subagents + two scripts: aggregate_benchmark, generate_review) and System B via one command (run_loop, which internally uses run_eval + improve_description + generate_report).
- Contract glue is **file-naming + JSON-schema conventions** documented in `references/schemas.md`; the viewer and aggregator both parse by exact field names and directory shapes. schemas.md explicitly warns that `config` vs `configuration` or unnested `pass_rate` renders as blank/zero.
- `history.json`, `metrics.json`, `comparison.json`, `analysis.json` schemas exist in schemas.md but no script reads/writes them — they are contracts for *Claude/subagents* to fulfill by hand.

## 4. Senior-engineer assessment

### Strengths
- **SKILL.md is genuinely excellent prompt engineering**: the conviction-over-compliance frame, keep/cut table, gotcha doctrine, progressive-disclosure-as-optimum, and self-demonstrating structure. It is the most reusable single artifact in the repo.
- **Trigger-eval mechanism is clever and real**: planting a temp command file and watching stream-json `input_json_delta` for early trigger detection (`run_eval.py:129-171`) measures the actual production trigger path, not a simulation. Early-exit on first non-Skill/Read tool keeps cost low. Env `CLAUDECODE` strip enables nesting.
- **Real ML hygiene in run_loop.py**: stratified holdout, blinding test scores from the improver (`run_loop.py:194-198`), best-by-test selection, per-query retry with trigger-rate thresholds, seed-fixed split, precision/recall reporting, per-iteration improver transcripts for debuggability.
- **Zero-dependency viewer**: stdlib-only server, self-contained HTML with base64-embedded outputs, static fallback for headless, regenerate-per-GET so refresh sees new runs, feedback protocol that distinguishes "no comment" from "not reviewed" (`viewer.html:1036-1043`).
- **Methodologically sound eval design**: paired same-turn with/without runs to cancel timing skew; grader that fails superficial compliance and critiques the evals themselves (grader.md Step 6); blind comparison to strip experimenter bias; analyzer forbidden from making improvement suggestions during benchmarking (role separation).
- **Human-in-the-loop is a first-class element**, not an afterthought: eval-set review UI, output review UI, feedback loop closing.

### Weaknesses / fragility
- **The System-A pipeline is convention-ware**: correctness depends on Claude hand-writing many JSONs (`grading.json`, `eval_metadata.json`, `timing.json`) with exact field names into an exact directory shape. Failure mode is *silent* (blank viewer cells, zero stats) — schemas.md itself has to warn about this. There is no schema validator script even though quick_validate.py shows they know how to write one.
- **Timing capture is inherently lossy**: `total_tokens`/`duration_ms` exist only in the subagent completion notification; forget to save and the datapoint is gone (SKILL.md §5, schemas.md timing.json). Fragile-by-design and depends on model discipline mid-session.
- **`aggregate_benchmark.py` delta is order-dependent**: "first two configs" by dict insertion order derived from sorted directory names (`load_run_results` iterates `sorted(eval_dir.iterdir())`) — `old_skill` sorts before `with_skill`, `with_skill` sorts after `without_skill`... actually `with_skill` < `without_skill` lexically works, but `new_skill` vs `old_skill` gives new<old — the sign of the delta silently flips depending on config *names*. SKILL.md papers over this with "put each with_skill config before its baseline."
- **Tokens metric is incoherent**: falls back to `output_chars` as a "token proxy" (`aggregate_benchmark.py:152-153`), so the tokens column can mix real token counts and character counts across runs in the same table.
- **Hand-rolled frontmatter parsing in `utils.py`** (line-prefix matching) vs `quick_validate.py` using real YAML — two parsers, two behaviors; utils breaks on flow-style or unusual-but-valid YAML.
- **`run_eval.py` mutates the user's project** (`.claude/commands/` file creation/deletion) — parallel workers each write/delete their own uuid'd file, but a crash between write and finally leaks files into the repo; also assumes cwd is inside the project.
- **Trigger detection heuristic**: counts a `Read` of the command file as a trigger and returns False the instant any other tool starts — a model that greps first or reads another file before invoking the skill is scored as non-triggering.
- **Comparator rubric pseudo-precision**: 1-5 Likert scores averaged to a decimal and scaled to 1-10 (comparator.md) — false quantification for what is a single LLM judgment; N=1, no position-swap counterbalancing (A/B ordering bias unaddressed).
- **Stats theater at N=3**: mean ± stddev on 3 runs per configuration; the viewer renders it authoritatively.
- **Dead/unwired schemas**: `history.json`, `metrics.json`, `comparison.json`, `analysis.json` are specified but nothing programmatic produces or consumes them — pure convention weight.
- **Viewer external CDN deps**: Google Fonts + SheetJS from cdn.sheetjs.com (`viewer.html:7-10`) — breaks offline and clashes with strict-CSP hosting.
- **Port management via `lsof`+SIGTERM of whatever holds :3117** (`generate_review.py:288-306`) — kills unrelated processes that happen to hold the port.
- **Base64-embedding all outputs** into one HTML page will bloat badly for large artifacts (videos, big PDFs, many runs).
- **`quick_validate.py` bans angle brackets in descriptions** — plausibly stale relative to current spec, and it rejects informative descriptions using `<placeholders>`.
- Minor: `improve_description.py` `test_results` param exists but run_loop never passes it (blinding makes it moot); emoji in package_skill output; `__pycache__` committed.

### What's overbuilt vs load-bearing
- Load-bearing and proven: SKILL.md, run_eval/run_loop/improve_description/generate_report (System B is fully automated end-to-end), generate_review + viewer.
- Overbuilt for most uses: blind comparator + post-hoc analyzer (SKILL.md itself admits "most iterations won't call for it"), the claims-extraction layer of the grader, four dead schemas, xlsx inline rendering.

## 5. Directly reusable for a harness-creator (CLAUDE.md + skills + hooks)

**Reusable nearly verbatim:**
- **SKILL.md §2 (conviction over compliance, keep/cut table, gotcha doctrine)** — applies with zero changes to CLAUDE.md authoring; arguably applies *more* since CLAUDE.md is always-in-context and every non-contributing line costs on every turn.
- **§3 progressive-disclosure economics** — maps directly onto the harness layering question: CLAUDE.md (always loaded) vs skills (load on trigger) vs references (load on reach) vs hooks (never in context — enforced deterministically). The "needed on every path?" test becomes the router for *which harness layer a given instruction belongs in*. §8's session-scoped-hooks paragraph is already the seed of the hooks story: "opinionated safety you'd never want globally" = the principle for when a rule should be a hook instead of prose.
- **The iterate-with-baseline eval loop shape** (§5-6): spawn paired subagent runs (with-harness vs without/old-harness), same-turn to cancel timing skew, snapshot-before-edit, descriptive eval names, workspace/iteration directory convention, transcripts-not-just-outputs, gotcha accumulation. All harness-agnostic — a "harness" is just a bigger unit under test than a skill.
- **`eval-viewer/` wholesale** (generate_review.py + viewer.html): it discovers "runs" purely by `outputs/` directories and renders grading/benchmark JSONs — nothing skill-specific except the word "skill" in labels and the `with_skill|without_skill|new_skill|old_skill` badge regex (`viewer.html:716`). Rename configs, done.
- **`agents/grader.md`, `agents/comparator.md`, `agents/analyzer.md`**: fully harness-agnostic already — they grade transcripts/outputs against expectations and compare outputs blind; only cosmetic "skill" wording needs changing. The grader's "critique the evals themselves" and the analyzer's discriminating-assertion analysis transfer untouched.
- **`aggregate_benchmark.py`**: config-name agnostic already (discovers dirs dynamically); needs only the delta-ordering fix and label changes.
- **`references/schemas.md` structure** as the pattern for pipeline contracts (though fix the dead schemas and add validation).
- **The human-review UX patterns**: eval-set editor (assets/eval_review.html), feedback.json protocol with empty-vs-unreviewed distinction, previous-iteration diffing.

**Reusable with adaptation:**
- **System B (trigger optimization)** adapts to skill-description tuning *within* a harness (harnesses contain skills, each needing trigger tuning) — run_eval/run_loop/improve_description work as-is for that sub-problem. The train/test-split + blinded-improver + best-by-holdout *pattern* also generalizes to optimizing CLAUDE.md content against behavioral evals, but the trigger-detection mechanism itself (command-file planting) is skill-specific; a CLAUDE.md/hook analogue needs a different measurable (e.g., did the hook fire, did the behavior comply).
- **`quick_validate.py`/`package_skill.py`**: the validate-then-package idea generalizes to harness linting (CLAUDE.md exists, hooks JSON parses, settings.json valid, skills validate) and harness packaging/distribution.
- **`improve_description.py`'s prompt architecture** (failures + history + "don't repeat, go structurally different" + hard limits + anti-overfitting framing) is the template for any LLM-in-the-loop optimizer over harness text.

**New ground a harness-creator must cover that skill-creator doesn't:**
- Hooks: deterministic enforcement vs persuasion (when to move a rule out of prose into a hook), hook testing (did it fire/block correctly), settings.json/permissions surface.
- CLAUDE.md: always-in-context budget pressure, project-vs-user-vs-local layering, interaction with skills' descriptions.
- Multi-component evals: a harness eval exercises trigger + instructions + hooks together; the eval unit is a session/task, not one prompt→output pair.
- Cross-component routing: which layer (CLAUDE.md line, skill, reference, hook, permission) should own a given behavior — §3's optimum logic is the seed but needs the hook/permission dimensions added.
