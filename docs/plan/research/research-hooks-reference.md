## 1. HOOK EVENTS (complete — 30 events)

> [계획 세션 정정] 원문은 "31 events"라 썼으나 1차 소스(07-hooks-reference.md)의 이벤트 수는 정확히 30개다. 아래 열거가 전수(30개)이며 Setup도 그 안에 포함된다.

Cadence classes: once/session (`SessionStart`, `SessionEnd`), once/turn (`UserPromptSubmit`, `Stop`, `StopFailure`), per-tool-call (`PreToolUse`, `PostToolUse`), plus standalone/async events.

### Common input fields (all events, JSON on stdin for command hooks / POST body for HTTP hooks)
`session_id`, `prompt_id` (v2.1.196+, matches OTel `prompt.id`), `transcript_path`, `cwd`, `permission_mode` ("default"|"plan"|"acceptEdits"|"auto"|"dontAsk"|"bypassPermissions"; "Manual" mode arrives as "default"; not on all events), `effort` ({level: low|medium|high|xhigh|max}; only on tool-context events), `hook_event_name`. Inside subagents/`--agent`: also `agent_id`, `agent_type`. Only SessionStart may receive `model` (not guaranteed). No `$CLAUDE_MODEL` env var exists.

### Per-event details

**SessionStart** — session begins/resumes. Matchers: `startup`, `resume`, `clear`, `compact`. Input: `source`, optional `model`, `agent_type`, `session_title`. Output: plain stdout IS added as context; JSON `hookSpecificOutput`: `additionalContext`, `initialUserMessage` (creates first user turn in `-p` mode), `sessionTitle` (only on startup/resume), `watchPaths` (array of absolute paths for FileChanged), `reloadSkills` (re-scan skill dirs so hook-installed skills work same session). No blocking; exit 2 = stderr shown as hook-error notice (v2.1.199+; earlier debug-log only). Has `CLAUDE_ENV_FILE` access (write `export` statements to persist env vars into all subsequent Bash commands). Only `command` and `mcp_tool` hook types. Re-runs on resume with source="resume" (can refresh stale context). MCP servers typically not yet connected when it fires.

**Setup** — only on `claude --init-only`, or `--init`/`--maintenance` with `-p`. Matchers: `init`, `maintenance`. Input: `trigger`. Cannot block. `additionalContext` via JSON (plain stdout goes to debug log only, unlike SessionStart). Has `CLAUDE_ENV_FILE`. Only `command`/`mcp_tool` types.

**InstructionsLoaded** — CLAUDE.md or `.claude/rules/*.md` loaded (eager at session start + lazy loads). Matchers: `session_start`, `nested_traversal`, `path_glob_match`, `include`, `compact`. Input: `file_path`, `memory_type` (User|Project|Local|Managed), `load_reason`, `globs`, `trigger_file_path`, `parent_file_path`. No decision control; async observability only; exit code ignored.

**UserPromptSubmit** — before Claude processes a submitted prompt. No matcher. Input: `prompt`. Default timeout lowered to 30s (command/http/mcp_tool). Output: plain stdout added as context (visible in transcript); JSON: `decision:"block"` (rejects+erases prompt), `reason` (shown to user, not added to context), `hookSpecificOutput.additionalContext` (invisible system-reminder), `sessionTitle`, `suppressOriginalPrompt` (with block). Exit 2 = block prompt. CANNOT replace/rewrite the prompt — only inject context alongside or block. Timed-out hook's output (incl. additionalContext) discarded; prompt proceeds without it.

**UserPromptExpansion** — when user-typed slash command/MCP prompt expands, before reaching Claude. Matcher: command name. Input: `expansion_type` (slash_command|mcp_prompt), `command_name`, `command_args`, `command_source`, `prompt`. Output: `decision:"block"`, `reason`, `additionalContext`. Covers path PreToolUse misses: typing `/skillname` directly bypasses the Skill tool's PreToolUse.

**MessageDisplay** — while assistant text streams (per batch of completed lines; in `-p`/SDK mode once per message with full text, index=0, final=true). No matcher. Default timeout 10s. Input: `turn_id`, `message_id` (not API msg_ id), `index`, `final`, `delta`. Output: `hookSpecificOutput.displayContent` replaces rendered text ON SCREEN ONLY — transcript and what Claude sees keep the original. Cannot block. Fires only for messages with text.

**PreToolUse** — before a tool call executes; can block/rewrite. Matcher: tool name (Bash, Edit, Write, Read, Glob, Grep, Agent, WebFetch, WebSearch, AskUserQuestion, ExitPlanMode, `mcp__<server>__<tool>`). Input: `tool_name`, `tool_input` (per-tool schema documented: Bash{command,description,timeout,run_in_background}; Write{file_path,content}; Edit{file_path,old_string,new_string,replace_all}; Read{file_path,offset,limit}; Glob{pattern,path}; Grep{pattern,path,glob,output_mode,-i,multiline}; WebFetch{url,prompt}; WebSearch{query,allowed_domains,blocked_domains}; Agent{prompt,description,subagent_type,model}; AskUserQuestion{questions,answers}; ExitPlanMode{plan,planFilePath,allowedPrompts — plan/planFilePath injected by Claude Code}), `tool_use_id`. Output (`hookSpecificOutput`): `permissionDecision` ("allow"=skip prompt / "deny"=block, reason shown to Claude / "ask"=prompt user with source label [User]/[Project]/[Plugin]/[Local] / "defer"= -p mode only, v2.1.89+, exits with stop_reason:"tool_deferred" + deferred_tool_use payload for SDK wrappers to resume; single-tool-call turns only), `permissionDecisionReason`, `updatedInput` (replaces ENTIRE tool input object before execution — rewrite/redact capability), `additionalContext`. Multi-hook precedence: deny > defer > ask > allow. Exit 2 = block. Deprecated top-level decision "approve"/"block" map to allow/deny. IMPORTANT: `@`-file references in prompts bypass all tool calls — no PreToolUse(Read) fires; use Read deny permission rules for that. `"allow"` cannot skip approval for MCP tools marked `_meta["anthropic/requiresUserInteraction"]` (v2.1.199+). AskUserQuestion/ExitPlanMode in `-p` mode need `"allow"`+`updatedInput` (with `answers` object) to proceed non-interactively.

**PermissionRequest** — when a permission dialog is about to be shown. Matcher: tool name. Input: `tool_name`, `tool_input` (no `tool_use_id`), optional `permission_suggestions` (array of would-be "always allow" options). Output (`hookSpecificOutput.decision`): `behavior` allow/deny, `updatedInput` (allow only; re-evaluated against deny/ask rules), `updatedPermissions` (allow only; array of permission-update entries: addRules/replaceRules/removeRules{rules:[{toolName,ruleContent?}],behavior}, setMode{mode}, addDirectories/removeDirectories{directories}; each with `destination`: session|localSettings|projectSettings|userSettings — hooks can PERSIST permission rules to settings files), `message` (deny), `interrupt` (deny, stops Claude). Exit 2 = deny. Does NOT fire in `-p` non-interactive mode — use PreToolUse there. setMode bypassPermissions only works if bypass already available at launch; never persisted as defaultMode.

**PermissionDenied** — auto-mode classifier denies a tool call (NOT manual denials, NOT hook blocks, NOT deny rules). Matcher: tool name. Input: `tool_name`, `tool_input`, `tool_use_id`, `reason`. Output: `hookSpecificOutput.retry: true` tells model it may retry (denial itself not reversed). Exit code/stderr ignored.

**PostToolUse** — after tool succeeds. Matcher: tool name. Input: `tool_name`, `tool_input`, `tool_response` (structured Output object, e.g. Write→{filePath,success}; Agent→{status,agentId,content,resolvedModel,totalTokens,totalDurationMs,totalToolUseCount,usage}; ExitPlanMode→{plan,filePath}), `tool_use_id`, `duration_ms`. Output: `decision:"block"`+`reason` (feedback to Claude; original output still visible), `additionalContext`, `updatedToolOutput` (REPLACES what Claude sees; must match tool's output schema for built-ins, else ignored; MCP outputs unvalidated), `updatedMCPToolOutput` (MCP only, prefer updatedToolOutput). Exit 2 = shows stderr to Claude but tool already ran (can't undo). Redaction pattern: PreToolUse.updatedInput for outbound, PostToolUse.updatedToolOutput for inbound.

**PostToolUseFailure** — after tool fails. Matcher: tool name. Input adds `error`, `is_interrupt`, `duration_ms`. Output: `additionalContext` only. Exit 2 = shows stderr to Claude.

**PostToolBatch** — once after a full parallel batch resolves, before next model call. No matcher. Input: `tool_calls` array [{tool_name, tool_input, tool_use_id, tool_response}] — here tool_response is the SERIALIZED tool_result content the model sees (different shape from PostToolUse). Output: `additionalContext`; `decision:"block"` or `continue:false` stops the agentic loop before next model call. Exit 2 = stops loop.

**Notification** — Claude Code sends a notification. Matchers: `permission_prompt`, `idle_prompt`, `auth_success`, `elicitation_dialog`, `elicitation_complete`, `elicitation_response`, `agent_needs_input` (v2.1.198+, agent-view only), `agent_completed` (same). Input: `message`, `title`, `notification_type`. Side effects only; cannot block/modify. Exit 2 = stderr to user only.

**SubagentStart** — subagent spawned. Matcher: agent type (`general-purpose`, `Explore`, `Plan`, custom frontmatter `name`, plugin-scoped `my-plugin:reviewer` — colon forces regex path so anchor `^…$`). Input: `agent_id`, `agent_type`. Output: `additionalContext` injected into SUBAGENT's context before first prompt. Cannot block subagent creation. Exit 2 stderr shows in subagent's transcript only.

**SubagentStop** — subagent finishes. Matcher: agent type. Input: `stop_hook_active`, `agent_id`, `agent_type`, `agent_transcript_path` (nested `subagents/` folder; `transcript_path` = main session), `last_assistant_message`, `background_tasks`, `session_crons` (parent-scoped, v2.1.145+). Output: same as Stop (`decision:"block"`+`reason` keeps subagent running with reason as next instruction; `additionalContext` = non-error feedback). Exit 2 = prevents subagent stopping. To inject into PARENT after subagent returns: use PostToolUse on Agent tool instead.

**TaskCreated / TaskCompleted** — task created via TaskCreate / marked completed (via TaskUpdate, or teammate finishing turn with in-progress tasks). No matchers. Input: `task_id`, `task_subject`, optional `task_description`, `teammate_name`, `team_name` (deprecated). Exit 2 = rolls back creation / prevents completion, stderr fed to model. JSON `{"continue":false,"stopReason":"..."}` stops the teammate entirely. Use case: enforce naming conventions, require passing tests before task close.

**Stop** — main agent finishes responding (not on user interrupt; API errors → StopFailure). No matcher. Input: `stop_hook_active` (true when already continuing due to a stop hook — MUST check to avoid loops), `last_assistant_message`, `background_tasks` [{id,type(shell|subagent|monitor|workflow|teammate|cloud session|MCP task),status,description,command,agent_type,server,tool,name}], `session_crons` [{id,schedule,recurring,prompt}] (v2.1.145+, distinguish "done" vs "paused for background work"). Output: `decision:"block"`+`reason` (required) prevents stopping, reason = Claude's next instruction; OR `hookSpecificOutput.additionalContext` = non-error feedback (continues conversation, labeled "Stop hook feedback", no error notice). Both go through loop protections: 8-consecutive-block cap (override via `CLAUDE_CODE_STOP_HOOK_BLOCK_CAP`, 0=disable). Exit 2 = prevent stopping. `/goal` command = built-in session-scoped prompt-based Stop hook.

**StopFailure** — turn ends due to API error. Matchers: `rate_limit`, `overloaded`, `authentication_failed`, `oauth_org_not_allowed`, `billing_error`, `invalid_request`, `model_not_found`, `server_error`, `max_output_tokens`, `unknown`. Input: `error`, `error_details`, `last_assistant_message` (= the API error string). Output and exit code IGNORED entirely. Logging/alerting only.

**TeammateIdle** — agent-team teammate about to go idle. No matcher. Input: `teammate_name`, `team_name` (deprecated). Exit 2 = teammate keeps working with stderr as feedback; `{"continue":false,"stopReason"}` stops teammate.

**ConfigChange** — config file changes mid-session. Matchers: `user_settings`, `project_settings`, `local_settings`, `policy_settings`, `skills`. Input: `source`, `file_path`. Output: `decision:"block"`+`reason` or exit 2 prevents the change applying to running session — EXCEPT `policy_settings` (managed) which can never be blocked (audit only).

**CwdChanged** — working dir changes (e.g. Claude runs `cd`). No matcher. Input: `old_cwd`, `new_cwd`. Output: `watchPaths` (replaces dynamic FileChanged watch list). No blocking. Has `CLAUDE_ENV_FILE` (direnv pattern: `direnv export bash > "$CLAUDE_ENV_FILE"`).

**FileChanged** — watched file changes on disk. Matcher is DUAL-PURPOSE: split on `|` into LITERAL filenames to watch (regex not useful for watch-list building; `.envrc|.env` watches exactly those), also filters which groups run against changed file's basename. Input: `file_path`, `event` (change|add|unlink). Output: `watchPaths`. No blocking. Has `CLAUDE_ENV_FILE`. Watch paths seeded from matcher + SessionStart/CwdChanged `watchPaths` returns.

**WorktreeCreate** — worktree being created (`--worktree` or subagent `isolation:"worktree"`). REPLACES default git-worktree behavior (enables SVN/Perforce/Hg). Input: `name` (slug). Output: command hook prints absolute worktree path on stdout (redirect everything else to stderr); HTTP hook returns `hookSpecificOutput.worktreePath`. ANY non-zero exit (not just 2) or missing path = creation fails. `.worktreeinclude` NOT processed when hook is configured — copy `.env` etc. yourself.

**WorktreeRemove** — worktree removed (session exit or subagent finish). Input: `worktree_path`. No decision control; failures logged in debug mode only.

**PreCompact** — before compaction. Matchers: `manual` (/compact), `auto`. Input: `trigger`, `custom_instructions` (manual only). Exit 2 or `decision:"block"` blocks compaction (for auto: if proactive, conversation continues uncompacted; if recovering from context-limit API error, the error surfaces and the request fails).

**PostCompact** — after compaction. Matchers: `manual`, `auto`. Input: `trigger`, `compact_summary`. No decision control.

**SessionEnd** — session terminates. Matchers (reason): `clear`, `resume`, `logout`, `prompt_input_exit`, `bypass_permissions_disabled`, `other`. Input: `reason`. No decision control. Default timeout 1.5s (!), auto-raised to highest per-hook settings timeout up to 60s (plugin timeouts don't raise it); override via `CLAUDE_CODE_SESSIONEND_HOOKS_TIMEOUT_MS`.

**Elicitation** — MCP server requests user input. Matcher: MCP server name. Input: `mcp_server_name`, `message`, `mode` (form|url), `url`, `elicitation_id`, `requested_schema`. Output: `hookSpecificOutput.action` (accept|decline|cancel) + `content` (form values) — answers programmatically, skipping the dialog. Exit 2 = deny.

**ElicitationResult** — after user responds, before response returns to server. Matcher: MCP server name. Input: `mcp_server_name`, `action`, `content`, `mode`, `elicitation_id`. Output: override `action`/`content`. Exit 2 = block (becomes decline).

## 2. CONFIGURATION FORMAT

Structure: `{"hooks": {"<EventName>": [{"matcher": "...", "hooks": [<handler>, ...]}, ...]}}`. Three nesting levels: event → matcher group → handler array. Matcher optional/ignored on no-matcher events (silently).

### Locations & scope
`~/.claude/settings.json` (user, all projects) / `.claude/settings.json` (project, committable) / `.claude/settings.local.json` (project, gitignored) / managed policy settings (org) / plugin `hooks/hooks.json` (optional top-level `description`; also inline in plugin.json or via `hooks` path field) / skill or agent YAML frontmatter (`hooks:` key; scoped to component lifetime, cleaned up when it finishes; `once: true` honored only here; subagent `Stop` auto-converts to `SubagentStop`; plugin-shipped AGENTS may NOT define hooks). No parent-directory fallback for settings.json hooks (unlike commands). Settings file watcher hot-reloads hook edits (ConfigChange fires).

### Matcher evaluation rules
- `"*"`, `""`, omitted → match all.
- Only letters/digits/`_`/`-`/space/`,`/`|` → exact string or `|`/`,`-separated list of exact strings. (`,` and whitespace tolerance v2.1.191+; hyphens in exact-set v2.1.195+ — earlier, `code-reviewer` was unanchored regex.)
- Any other char → unanchored JS RegExp (`RegExp.test`); `Edit.*` matches NotebookEdit too — anchor `^Edit$` for whole-string.
- `FileChanged`/`StopFailure` use narrower exact set (letters/digits/`_`/`|` only; only `|` separates).
- MCP tools: `mcp__<server>__<tool>`; `mcp__memory__.*` for all server tools (the `.*` is REQUIRED — bare `mcp__memory` is exact-match and matches nothing).

### Handler types (5)
1. **command**: fields `command` (required), `args` (presence switches to EXEC FORM: command resolved on PATH, spawned directly, no shell, args passed verbatim, placeholders substituted as plain strings — preferred for path placeholders; absence = SHELL FORM: `sh -c` on mac/Linux, Git Bash or PowerShell on Windows), `async` (background, non-blocking, decisions ignored; `additionalContext` delivered next turn; suppressed completion notices unless verbose), `asyncRewake` (implies async; exit 2 wakes Claude immediately with stderr/stdout as system reminder), `shell` ("bash"|"powershell"; ignored with args). Windows: exec form needs real .exe (npm .cmd shims fail — use `node` + script path).
2. **http**: `url` (required), `headers` (env interpolation `$VAR`/`${VAR}` gated by `allowedEnvVars`), `allowedEnvVars`. POST JSON body; response 2xx+JSON = decision, 2xx+text = context, non-2xx/timeout/connection failure = NON-blocking error (cannot block via status code — must return 2xx with decision JSON).
3. **mcp_tool**: `server` (must already be connected — no OAuth/connect flow triggered; SessionStart/Setup usually fire before connect), `tool`, `input` (values support `${path}` substitution from hook input, e.g. `"${tool_input.file_path}"`). Tool text output treated like command stdout. `isError:true` → non-blocking error.
4. **prompt**: `prompt` (required; `$ARGUMENTS` placeholder for input JSON, appended if absent; `\$` escapes), `model` (default fast/Haiku), `continueOnBlock`. Single LLM call returns `{"ok": true|false, "reason"}`. Per-event ok:false behavior: Stop/SubagentStop→reason fed back, continue; PreToolUse→deny (=permissionDecision deny); PostToolUse→turn ends w/ warning (continueOnBlock:true feeds back instead); PostToolBatch/UserPromptSubmit/UserPromptExpansion→turn ends regardless; PostToolUseFailure/TaskCreated/TaskCompleted→tool-error to Claude; TeammateIdle→teammate stops (continueOnBlock keeps working); PermissionRequest→NO EFFECT; PermissionDenied→NO EFFECT (can't set retry).
5. **agent** (EXPERIMENTAL): same fields as prompt; spawns subagent with Read/Grep/Glob etc., up to 50 turns, default timeout 60s, same ok/reason schema. For verifications requiring actual file/test inspection.

Type support by event: all five → PermissionDenied, PermissionRequest, PostToolBatch, PostToolUse, PostToolUseFailure, PreToolUse, Stop, SubagentStop, TaskCompleted, TaskCreated, TeammateIdle, UserPromptExpansion, UserPromptSubmit. command/http/mcp_tool only → ConfigChange, CwdChanged, Elicitation, ElicitationResult, FileChanged, InstructionsLoaded, Notification, PostCompact, PreCompact, SessionEnd, StopFailure, SubagentStart, WorktreeCreate, WorktreeRemove. command/mcp_tool only → SessionStart, Setup.

### Common handler fields
`type` (required), `if` (ONE permission rule, e.g. `"Bash(git *)"`, `"Edit(*.ts)"`; v2.1.85+; tool events ONLY — on other events a hook with `if` NEVER runs; no &&/||; Bash matching checks each subcommand, `$()`, backticks; leading VAR=val stripped; args-specifying patterns fail-open on `$()`/backticks/`$VAR`; unparseable commands fail-open → `if` is BEST-EFFORT, not a security boundary), `timeout` (seconds; defaults 600 command/http/mcp_tool, 30 prompt, 60 agent; UserPromptSubmit→30, MessageDisplay→10, SessionEnd budget 1.5s), `statusMessage` (custom spinner text), `once` (skill frontmatter only).

### Execution semantics
All matching hooks run IN PARALLEL; identical handlers deduplicated (command hooks by command+args, HTTP by URL). One hook's deny does NOT stop siblings' side effects. Results merged: PreToolUse most-restrictive wins (deny>defer>ask>allow); all `additionalContext` values concatenated. Multiple PreToolUse `updatedInput`: LAST to finish wins (non-deterministic — avoid). Path placeholders `${CLAUDE_PROJECT_DIR}`, `${CLAUDE_PLUGIN_ROOT}`, `${CLAUDE_PLUGIN_DATA}` substituted AND exported as env vars; plugins also `${user_config.*}` + `CLAUDE_PLUGIN_OPTION_<KEY>` env.

## 3. EXIT CODE / JSON OUTPUT CONTRACT

- **Exit 0**: success; stdout parsed for JSON decision fields (JSON processed ONLY on exit 0). Stdout goes to debug log except UserPromptSubmit/UserPromptExpansion/SessionStart where it becomes context.
- **Exit 2**: blocking error; stdout/JSON IGNORED; stderr fed to Claude (per-event effect table above). Mutually exclusive with JSON control — pick one per hook.
- **Exit 1 / other**: NON-blocking error; action proceeds; transcript shows `<hook> hook error` + first stderr line. GOTCHA: exit 1 does NOT block despite Unix convention. Exception: WorktreeCreate fails on ANY non-zero.
- **Universal JSON fields**: `continue:false` (stops Claude entirely, precedence over everything) + `stopReason` (user-facing, not shown to Claude), `suppressOutput`, `systemMessage` (user-facing warning), `terminalSequence` (v2.1.141+; allowlisted OSC 0/1/2/9/99/777 + BEL only — desktop notifications/titles/bell; /dev/tty unavailable to hooks since v2.1.139).
- **Output caps**: all hook output strings (additionalContext, systemMessage, stdout) capped at 10,000 chars; overflow saved to file + preview/path substituted.
- **additionalContext**: wrapped in system reminder, invisible in chat; placement varies by event (session start / alongside prompt / next to tool result / end of turn). Phrase as FACTUAL statements ("This repo uses bun test") not imperative system commands — imperative out-of-band phrasing can trip prompt-injection defenses and get surfaced to the user instead. On `--resume`/`--continue`, mid-session injected text is REPLAYED not re-run (timestamps/SHAs go stale); SessionStart re-runs with source="resume".
- **Decision patterns by event**: top-level `decision:"block"`+`reason` → UserPromptSubmit, UserPromptExpansion, PostToolUse, PostToolUseFailure, PostToolBatch, Stop, SubagentStop, ConfigChange, PreCompact. `hookSpecificOutput` → PreToolUse (permissionDecision), PermissionRequest (decision.behavior), PermissionDenied (retry), Elicitation/ElicitationResult (action/content), MessageDisplay (displayContent). Exit-code/continue → TeammateIdle, TaskCreated, TaskCompleted. Path-return → WorktreeCreate. Context-only → SessionStart, Setup, SubagentStart. None → WorktreeRemove, Notification, SessionEnd, PostCompact, InstructionsLoaded, StopFailure, CwdChanged, FileChanged.

## 4. WHAT HOOKS CAN GUARANTEE vs CANNOT

### CAN enforce deterministically
- Block tool calls before execution (PreToolUse deny/exit 2) — works EVEN IN bypassPermissions/`--dangerously-skip-permissions` mode; users cannot bypass by changing permission mode. Blocking hook also beats allow rules (pattern: allow-all Bash + hook-deny specific commands).
- Block edits to protected files (PreToolUse Edit|Write matcher + path check, exit 2 with stderr feedback so Claude adapts).
- Rewrite tool inputs before execution (updatedInput — redact secrets, rewrite commands) and tool outputs before Claude sees them (updatedToolOutput — must match built-in output schema or it's ignored).
- Force actions after edits (PostToolUse Edit|Write → prettier/lint; async:true for slow suites with results fed back via additionalContext).
- Inject context: session start (stdout or additionalContext), post-compaction re-injection (SessionStart matcher "compact"), per-prompt (UserPromptSubmit), per-tool-result, into subagents (SubagentStart), batch-level (PostToolBatch).
- Validate-before-stop / keep-Claude-working loops (Stop decision:block with stop_hook_active guard + 8-block cap; prompt/agent hooks for judgment calls; TaskCompleted/TeammateIdle gates for teams).
- Auto-approve/deny permission dialogs (PermissionRequest), persist permission rules/modes to settings files (updatedPermissions + destination), tell auto-mode it may retry (PermissionDenied).
- Block prompts/slash-commands (UserPromptSubmit/UserPromptExpansion), block config changes (ConfigChange, except policy), block compaction (PreCompact), answer MCP elicitations programmatically, replace worktree VCS, persist env vars (CLAUDE_ENV_FILE from SessionStart/Setup/CwdChanged/FileChanged), rename sessions, watch files, desktop notifications (terminalSequence / Notification event), audit logging everywhere.

### CANNOT
- Undo PostToolUse effects (tool already ran; updatedToolOutput changes only what Claude sees; telemetry captured original).
- Rewrite the user's prompt (UserPromptSubmit: inject or block only).
- Block: session start/end, subagent start, notifications, file changes, cwd changes, compaction results, instruction loads, worktree removal, StopFailure (output fully ignored), policy_settings changes.
- Override deny/ask permission rules: hook "allow" never bypasses deny rules from ANY scope incl. managed settings (hooks tighten, never loosen). Managed deny lists always win.
- Trigger slash commands or tool calls from a command hook (stdout/stderr/exit only; additionalContext is plain text to Claude).
- See file changes made via Bash with an Edit|Write matcher (compensate: Stop hook scanning `git status --porcelain`, or also match Bash).
- See `@`-referenced files (no tool call happens; use Read deny rules).
- Rely on `if` as a hard gate (best-effort, fails open on unparseable commands — use permission system for hard allow/deny).
- Async hooks can't block or return decisions; PermissionRequest doesn't fire in `-p` mode; prompt/agent hooks can't affect PermissionRequest/PermissionDenied; MessageDisplay can't change transcript/model view.
- Guarantee ordering among parallel hooks (updatedInput races).

## 5. SECURITY / GOTCHAS / ENV

- Command hooks run with FULL user permissions, unsandboxed, in own session without controlling terminal (no /dev/tty). Best practices: quote variables, block `..` traversal, absolute paths, skip .env/.git/keys.
- Env available to hook processes: `CLAUDE_PROJECT_DIR`, `CLAUDE_PLUGIN_ROOT`, `CLAUDE_PLUGIN_DATA`, `CLAUDECODE=1`, `CLAUDE_CODE_CHILD_SESSION=1` (v2.1.172+), `CLAUDE_CODE_SESSION_ID`, `CLAUDE_EFFORT`, `CLAUDE_ENV_FILE` (4 events only), `$CLAUDE_CODE_REMOTE` ("true" in cloud), `$CLAUDE_CODE_BRIDGE_SESSION_ID` (v2.1.199+ w/ Remote Control), `TRACEPARENT` (opt-in).
- Enterprise controls (settings): `disableAllHooks` (respects managed hierarchy — can't disable managed hooks from lower scopes; no per-hook disable exists), `allowManagedHooksOnly` (managed-only; blocks user/project/plugin hooks except plugins force-enabled in managed enabledPlugins, trust by full plugin@marketplace ID), `strictPluginOnlyCustomization` (true or ["skills","hooks",...]), `allowedHttpHookUrls` (URL allowlist, `*` wildcard, empty array = block all HTTP hooks, merges across sources), `httpHookAllowedEnvVars` (intersected with per-hook allowedEnvVars). `--safe-mode`/`CLAUDE_CODE_SAFE_MODE=1` skips user hooks but keeps policy hooks. `CLAUDE_CODE_SUBPROCESS_ENV_SCRUB=1` strips API credentials from hook subprocesses.
- Shell-profile stdout pollution breaks JSON parsing (guard profile echoes with `[[ $- == *i* ]]`).
- PowerShell: `shell:"powershell"`; `${CLAUDE_PROJECT_DIR}` rewritten to `${env:NAME}` in shell-form as of v2.1.198 (earlier: plugin hooks only; use `$env:` form); bare `$CLAUDE_PROJECT_DIR` resolves to $null in PowerShell.
