## 1. Protected paths — critical omission (source: `01-getting-started/05-use-claude-code/02-permission-modes.md` §Protected paths, v2.1.160+)
Writes to a fixed path set are NEVER auto-approved in any mode except `bypassPermissions`, and `permissions.allow` rules in settings do NOT pre-approve them (the safety check runs BEFORE allow-rule evaluation — an `Edit(.claude/**)` allow rule changes nothing). Per mode: default/acceptEdits/plan → prompt; auto → routed to classifier; dontAsk → denied. Protected directories: `.git`, `.config/git`, `.vscode`, `.idea`, `.husky`, `.cargo`, `.devcontainer`, `.yarn`, `.mvn`, and **`.claude` (except `.claude/worktrees`)**. Protected files include `.mcp.json`, `.claude.json`, shell rc files (`.zshrc`, `.bashrc`, `.profile`, `.envrc`...), `.npmrc`/`.yarnrc`/`bunfig.toml`, `.bazelrc`, `.pre-commit-config.yaml`/`lefthook.yml`, `gradle-wrapper.properties`, `.gitconfig`/`.gitmodules`, `.devcontainer.json`, `.ripgreprc`, `pyrightconfig.json`. In prompting modes, the first `.claude/` write prompt offers "Yes, and allow Claude to edit its own settings for this session" which approves subsequent `.claude/` writes for that session. Direct consequence for the harness-creator skill itself: every generated artifact it writes (`.claude/skills/`, `.claude/settings.json`, `.claude/agents/`, `.mcp.json`) is a protected-path write — the plan must account for one unavoidable prompt + session-scoped approval, and must not claim allow rules can silence it.

## 2. Canonical tool list + default-permission column missing (source: `05-reference/04-tools-reference.md`)
A generator emitting `tools:`, `allowed-tools`, or permission rules needs the exact names. Full set: Agent, Artifact, AskUserQuestion, Bash, CronCreate, CronDelete, CronList, Edit, EnterPlanMode, EnterWorktree, ExitPlanMode, ExitWorktree, Glob, Grep, ListMcpResourcesTool, LSP, Monitor, NotebookEdit, PowerShell, PushNotification, Read, ReadMcpResourceTool, RemoteTrigger, ReportFindings, ScheduleWakeup, SendMessage, SendUserFile, ShareOnboardingGuide, Skill, TaskCreate, TaskGet, TaskList, TaskOutput (deprecated), TaskStop, TaskUpdate, TodoWrite (disabled by default since v2.1.142), ToolSearch, WaitForMcpServers, WebFetch, WebSearch, Workflow, Write. Tools that prompt by default ("Permission required: Yes"): Artifact, Bash, Edit, ExitPlanMode, Monitor, NotebookEdit, PowerShell, ShareOnboardingGuide, **Skill**, WebFetch, WebSearch, Workflow, Write. Rule-format cross-application: `Read(path)` rules also govern **Grep, Glob, LSP**; `Edit(path)` rules also govern **Write and NotebookEdit**; `Bash(cmd)` rules also govern **Monitor**. An `Edit(...)` allow rule implicitly grants Read on the same path. Subagent tool resolution: neither field → inherit all; `tools` only → only listed; `disallowedTools` only → all minus listed; both → disallowedTools wins.

## 3. Auto-mode facts that change what a generator should emit (source: `02-permission-modes.md`)
- On entering auto mode, broad allow rules are **dropped**: `Bash(*)`, `PowerShell(*)`, wildcarded interpreters (`Bash(python*)`), package-manager run commands, and all `Agent` allow rules. Narrow rules (`Bash(npm test)`) survive. Generated blanket allows are dead weight under auto mode.
- The classifier reads user messages, tool calls, and **CLAUDE.md content** (tool results stripped) — generated CLAUDE.md text directly shapes auto-mode allow/block decisions; boundaries stated in conversation block matching actions but can be lost to compaction (deny rules are the durable form).
- Subagent frontmatter `permissionMode` is **ignored** under auto mode.
- Fallback: 3 consecutive or 20 total classifier blocks pause auto mode; in `-p` mode repeated blocks **abort the session** (matters for scripted eval runs).
- `dontAsk` mode **denies** explicit `ask` rules rather than prompting; only allow-rule matches + built-in read-only Bash run.
- `acceptEdits` also auto-approves in-scope `mkdir, touch, rm, rmdir, mv, cp, sed` (and PowerShell `Set-Content` etc. when that tool is enabled).
- Cloud (Claude Code on the web) silently ignores `defaultMode: bypassPermissions|dontAsk` from checked-in settings.

## 4. CLAUDE.md mid-session staleness + cache layering (source: `01-getting-started/04-core-concepts/05-prompt-caching.md`)
- Project-root and user CLAUDE.md are read **once at session start**; mid-session edits do NOT apply until `/clear`, `/compact`, or restart (lines 154-156). SKILL.md edits DO apply live (skills doc). A harness-creator that edits CLAUDE.md mid-session must tell the user the running session won't see it — and its own eval loop must restart sessions between iterations to pick up CLAUDE.md changes.
- Skills, commands, agents, hooks, LSP servers, monitors, themes **never invalidate the prompt cache** (appended after conversation); MCP tool-list changes DO when tools are loaded into the prefix (tool search unavailable/disabled or `alwaysLoad`).
- Nested CLAUDE.md / `paths:`-scoped rules load lazily; edits before first load take effect, edits after don't.
- Cache layers: system prompt (tool defs; change invalidates everything) → project context (CLAUDE.md, auto memory, unscoped rules; re-read at start//clear//compact) → conversation. Model and effort level are separate cache keys.

## 5. Skill listing is NOT re-injected after /compact (source: `01-getting-started/04-core-concepts/04-context-window.md`, simulation data lines 56-57)
"Unlike the rest of the startup content, this listing is not re-injected after `/compact`. Only skills you actually invoked get preserved." — i.e., auto-invocation via descriptions degrades after compaction. Relevant to harness durability claims and long-session eval design.

## 6. `.claude/loop.md` — committable harness surface missing from all summaries (source: `02-build-with-claude-code/06-automation/03-run-prompts-on-a-schedule.md` lines 104-122)
`.claude/loop.md` (project, precedence) or `~/.claude/loop.md` (user) replaces the built-in maintenance prompt for bare `/loop`. Plain markdown, single default prompt. A generated harness can ship one. Related: session crons are min 1-minute, 7-day expiry, restored on `--resume` if unexpired; as of v2.1.196 a scheduled fire only runs skills Claude may invoke itself — skills with `disable-model-invocation: true`, `skillOverrides`-withheld skills, `Skill`-deny-ruled skills, and built-in commands arrive as plain text and do NOT execute (a generator pairing `disable-model-invocation` with `/loop` automation would silently break). No other schedule surface is repo-committable (Routines = cloud per-user; Desktop tasks = per-machine).

## 8. Correction: `--safe-mode` scope understated in the hooks summary
Source: `02-build-with-claude-code/10-troubleshooting/03-debug-configuration.md` line 70 (v2.1.169+). Safe mode disables ALL customizations — CLAUDE.md, skills, plugins, hooks, MCP servers, custom commands and agents — not just user hooks. Managed hooks/settings policy still apply; managed plugins/skills/CLAUDE.md/MCP are turned off. `CLAUDE_CONFIG_DIR` pointed at an empty dir bypasses everything under `~/.claude`. This is the canonical "is the generated harness causing the problem?" debugging tool a harness-creator should document.

## 9. /goal operational constraints (source: `02-build-with-claude-code/07-goals/01-goal.md`)
It's a wrapper around a session-scoped prompt-based Stop hook on the small-fast model (default Haiku). Requires the workspace trust dialog accepted; unavailable when `disableAllHooks` is set at ANY level or `allowManagedHooksOnly` in managed settings. Condition ≤4,000 chars; evaluator reads only the transcript (runs no commands) — conditions must be provable by Claude's own surfaced output. Works non-interactively: `claude -p "/goal <condition>"` runs the loop to completion (useful as an eval-harness primitive). One goal per session; restored on `--resume` (counters reset).

## 10. Sandbox↔permissions interaction (source: `04-configuration/01-settings-and-permissions/04-bash-sandbox.md`)
If a generated harness sets `sandbox.enabled`: auto-allow mode auto-approves sandboxed Bash regardless of permission mode (even outside acceptEdits); bare `Bash`/`Bash(*)` **ask** rules are skipped for sandboxed commands, but content-scoped ask rules (`Bash(git push *)`) still prompt; deny rules always hold. Escape hatch: Claude may retry failed commands with `dangerouslyDisableSandbox` (goes through normal permission flow) unless `allowUnsandboxedCommands: false`. `Edit(...)` allow rules grant sandbox write access identically to `sandbox.filesystem.allowWrite`; both sources merge. Sandboxed vs unsandboxed commands see different `$TMPDIR`.
