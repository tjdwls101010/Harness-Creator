# Agents

This is the authoring guide for `.claude/agents/*.md` — custom subagents. Read this before proposing any agent role during the interview, and re-check every generated agent definition against it before declaring the component done.

## The eligibility test: don't generate an agent by default

The temptation in a meta-skill like this one is to treat "agents" as a deliverable you're expected to produce — the interview surfaced some roles, so you write some agent files. Resist that. An agent is justified only when one of two things is true: **context isolation is actually valuable** (the work is read-heavy — research, code review, QA sweeps, log triage — and only the conclusion needs to survive back in the main thread, not the search trail that produced it), or **the task genuinely needs a distinct tool restriction or a distinct system prompt** that the main conversation shouldn't carry by default (a read-only database analyst, a reviewer that must never Edit). If neither condition holds, the right answer is no agent at all — just do the work in the main conversation, or write it as a skill that runs inline.

Agent count is itself a cost, independent of any single agent's quality. Every agent you add is one more role the orchestrating Claude has to keep in mind when deciding where to route a task, one more `description` competing for the same routing attention, one more definition that can silently rot out of sync with the codebase. A harness with twelve narrowly-scoped agents is not more capable than one with three well-chosen ones — it's more expensive to reason about and more likely to misroute. This project's own research turned up a cautionary precedent: a different developer's meta-harness (internally called revfactory) made "Agent Teams, 4-5 agents" the unconditional default architecture for every generated harness, regardless of what the target project actually needed — a travel-planning harness got the same five-agent shape as an incident-postmortem harness. The number was fixed before the domain was even known. Don't repeat that: generate a role only when the interview has produced a concrete, demonstrated need for it — a specific recurring task, a specific volume-of-output problem, a specific tool-restriction requirement — never as a default shape you reach for because "harnesses have agents."

Concretely, when the interview surfaces something that sounds agent-shaped, ask: would this task's exploration output (file contents, search results, log lines, diff context) be useless to the main conversation once a verdict is reached? If yes, that's real signal for isolation. Would the main conversation actively benefit from seeing the intermediate steps — because the user might want to redirect mid-task, or because the next step depends on details from this one? If yes, keep it in the main thread; an agent would just add a round-trip and a lossy summary. A security-focused code reviewer that reads a full diff and fifty lines of surrounding context per finding, then returns five sentences of verdict, is a clean case for isolation. A task like "rename this function and update its ten call sites" is not — there's no volume to isolate, and the main conversation needs to see each edit to catch a bad rename before it compounds.

## The body is a full replacement, not an addition

An agent's markdown body becomes its entire system prompt. Claude Code's default system prompt — the one that knows to be concise, to prefer editing over rewriting, to run tests before declaring done, all the baseline behavior you get in an ordinary session — is gone entirely when a custom agent runs. The subagent receives only what you write in the body, plus bare environment details (working directory and similar), plus whatever CLAUDE.md and skills content loads through the normal mechanisms. Nothing else carries over.

The practical implication: write generated agent bodies as if briefing someone who has never used Claude Code and has no idea what "the default way Claude behaves" even means. If the agent's job requires being concise, say so. If it needs to run a verification step before finishing, spell that out — don't assume "run tests before declaring done" survives from some ambient default, because for this agent there is no ambient default. A one-line body like "You are a code reviewer, review the code" is not a thin version of the system prompt — it's the *entire* system prompt, and everything else that normally scaffolds good behavior is simply absent for the duration of this agent's run.

## `model` defaults to inherit; forcing a tier needs a reason

Leave `model` unset (or set it to `inherit`) unless you have a specific, statable reason to pin a tier. Inherit means the agent runs on whatever model the parent conversation is using — the cost and capability tradeoff the user already chose for this session, carried through automatically. Forcing `opus` for every agent regardless of task, the way revfactory's harness generator did as a hard default, burns cost and latency on tasks that don't need the extra capability — routine log-triage or a lint-style review doesn't need the same tier as a subtle multi-file architecture judgment. Conversely there are legitimate reasons to pin a specific model: a high-volume, low-complexity role (classify these hundred log lines into three buckets) is a real case for pinning `haiku` to keep repeated invocations cheap and fast, and a role that makes a small number of unusually hard judgment calls (final security sign-off on a release) is a real case for pinning a stronger tier even if the parent session runs cheaper. The test is the same either way: state the reason next to the pin. "This agent runs on `haiku` because it's invoked per-file across large sweeps and the task is pattern-matching, not judgment" is a deliberate tradeoff. A blanket `model: opus` on every generated agent with no task-specific justification is the default-maximal antipattern to avoid.

## Gotcha: Explore and Plan don't load CLAUDE.md or git status at all

The two most commonly auto-invoked built-in subagents — Explore (fast read-only search) and Plan (research during plan mode) — skip the CLAUDE.md hierarchy and the parent session's git status entirely, by design, to keep exploration fast and cheap. This is not configurable per-agent; there's no frontmatter field that changes it. Every other built-in and every custom subagent you generate *does* load both. So if a harness rule genuinely needs to reach a delegated task — "ignore everything under `vendor/`," "never touch files in `legacy/`" — and that delegation happens to go through Explore or Plan, the rule in your generated CLAUDE.md simply never arrives. The main conversation sees Explore's or Plan's results with full CLAUDE.md context of its own, so most rules don't need to reach the subagent itself; the gap only matters for a rule the *subagent's own behavior* must obey while it's running, not a rule about how to interpret what it finds.

When the interview identifies a rule that must reach a delegated task, there are exactly three fixes, and picking the right one per case is a judgment call, not a lookup:

1. **Restate the rule directly in the delegation prompt text.** Since CLAUDE.md text doesn't propagate to Explore or Plan, put it in the sentence that requests the delegation instead — an instruction in the main harness surface telling Claude "when delegating searches to Explore, always tell it to skip `vendor/`" bakes the restatement into the ask itself. Cheapest fix, no new files, but only as durable as the phrasing of each delegation — easy to forget on an ad hoc request.
2. **Replace the built-in with a custom agent of the same name.** A project or user-scoped agent file named `Explore` overrides the built-in of the same name, and a custom agent — unlike the true built-in — does load CLAUDE.md and git status like any other custom subagent, and its body can additionally restate the critical rule outright. Use this when the rule needs to hold on *every* Explore-shaped delegation in the project, not just ones the main conversation remembers to caveat.
3. **Inject it via a `SubagentStart` hook's `additionalContext`.** A project-level hook matched on the agent type name fires when the subagent begins and can add context to its startup state programmatically, which reaches even the true built-ins (since the hook operates at the session level, not inside the agent definition). Use this when the rule is closer to "operational fact the subagent should know" than "constraint on the main conversation's phrasing," or when you want it enforced by configuration rather than by remembering to phrase things right in the moment.

Pick (1) for a one-off or low-stakes case, (2) when you're generating a project that leans on Explore/Plan heavily and the rule is load-bearing, and (3) when you want the fix to live in `settings.json` rather than in prompt discipline or an agent file — e.g., because the same `additionalContext` should also reach other subagent types uniformly.

## Gotcha: `skills:` preloads full skill bodies, not descriptions

An agent's `skills:` frontmatter field injects the **complete body** of every listed skill into that agent's context at startup — not just the skill's description, the way the main conversation's skill-discovery mechanism works. This is a real context-cost decision every time you add an entry, not a free convenience for "giving the agent some background." A 300-line skill listed here is 300 lines the agent carries into every single invocation, whether or not the current task actually needs that skill's content. Reserve `skills:` for content the agent needs on essentially every run of its job (a code-reviewer agent that always needs a project's `error-handling-conventions` skill loaded to judge against it) rather than content it might occasionally want — for the occasional case, leave the skill out of `skills:` and let the agent discover and invoke it on demand through the Skill tool instead, which only pays the cost when actually used.

## Gotcha: agent-scoped hooks and the `Stop`→`SubagentStop` conversion

Hooks declared inside an agent's own frontmatter (as opposed to `settings.json`) are scoped to that agent's lifetime — they run only while the agent is active and are cleaned up the moment it finishes, which makes them the right place for validation that's specific to one agent's job (a `PreToolUse` hook on `Bash` that blocks anything but `SELECT` for a read-only database agent, for instance) rather than a project-wide concern. One naming gotcha inside this mechanism: a `Stop` hook declared in an agent's frontmatter is automatically treated as `SubagentStop` at runtime. Write `Stop` in the agent file; Claude Code fires it when *that agent* stops, not when the main session stops — don't confuse this with a project-level `SubagentStop` hook in `settings.json`, which is the same semantic reached a different way (matched by agent-type name from the session level instead of declared inline).

## Gotcha: identity is `name`, not the filename, and duplicates fail silently

An agent's identifier — the thing Claude actually matches on for delegation, `@`-mention, and `Agent(name)` permission rules — is its frontmatter `name` field, never the filename. You can name the file `reviewer.md` and give it `name: code-reviewer` and everything resolves correctly by the frontmatter value alone; the filename is purely for humans browsing the directory. The dangerous consequence: if two agent files in the same scope (both project-level, say) declare the same `name`, Claude Code loads only one of them — silently, no error, no warning at generation time. If you generate `security-reviewer.md` and `sec-reviewer.md` and both declare `name: reviewer` by accident, one of them is simply invisible from that point on, and nothing in the normal flow tells you which. This is exactly the kind of failure your validation script should catch mechanically: lint every generated agent's frontmatter `name` field for uniqueness across the project scope before declaring the harness complete, because a human skimming filenames will not notice the collision.

## Gotcha: AskUserQuestion does not exist inside a subagent

`AskUserQuestion` is one of a small set of tools tied to the main conversation's UI and session state, and it is unavailable to any subagent regardless of what the `tools:` frontmatter field lists — listing it changes nothing, because the restriction isn't about permission, it's about the tool having nowhere to render an interactive prompt from inside an isolated agent context. This has two separate consequences for you as the generator of harness-creator's own behavior and as the generator of other agents:

First, it's why harness-creator's own interview stage must never be delegated to a subagent. The whole value of the interview is turning ambiguity into an explicit, structured choice the user makes — and that mechanism simply isn't reachable from inside a delegated task. An agent asked to "interview the user about their testing preferences" cannot do that; it would have to either fail, guess, or fall back to unstructured text output that the main conversation would then have to relay back and forth anyway, defeating the entire point of delegating.

Second, and just as binding for anything you generate for someone else's project: never design an agent whose job description assumes it can interactively ask the user things. If an interview answer describes a role like "an agent that asks the user clarifying questions about ambiguous requirements before proceeding," that role cannot exist as a subagent as specified — it has to stay in the main conversation (as a skill or as ordinary conversational flow), or the agent has to be redesigned to work from a fully-specified brief handed to it up front, with no expectation of a follow-up question mid-task.

## Frontmatter fields in practice

Only `name` and `description` are required. The rest is the full schema; this table covers only the fields that require a judgment call when you're generating an agent, not an exhaustive dump of every field that exists.

| Field | Judgment call |
|---|---|
| `tools` | Omit to inherit everything (simplest, right default for a general-capability agent). List explicitly as an allowlist when the whole point of the agent is a restriction — e.g. `Read, Grep, Glob, Bash` with no `Edit`/`Write` for a review-only role. Prefer this over `disallowedTools` when the safe set is small and enumerable; prefer `disallowedTools` when you want "everything except X" (e.g. inherit everything but deny `Write, Edit`) and the exception list is shorter than the inclusion list. |
| `model` | Default to unset/`inherit` (see above). Pin only with a stated reason — high-volume+low-complexity toward a cheaper tier, rare+high-stakes toward a stronger one. |
| `skills` | Only list a skill here if the agent needs its full body on essentially every invocation. Otherwise leave it out and let the agent discover it on demand via the Skill tool. |
| `permissionMode` | Leave unset (inherits the parent's mode) unless the agent's whole purpose is a stricter posture — e.g. a read-only analyst that should never even be asked to accept an edit prompt. Note the parent's `bypassPermissions`/`acceptEdits` always wins over whatever you set here; this field can tighten a `default`-mode parent, not loosen a locked-down one. |
| `hooks` | Only for validation specific to this one agent's job (see the `PreToolUse` read-only-database example above). Scoped to the agent's lifetime; remember `Stop` becomes `SubagentStop`. Don't use this for project-wide policy — that belongs in `settings.json`. |
| `isolation: worktree` | Set when the agent's file edits should land in a separate git worktree rather than the parent checkout — useful for an agent you want to experiment freely without risking the main working tree, at the cost of needing to merge or discard the worktree afterward. |
| `color` | Cosmetic only (transcript/task-list display). Fine to omit; only worth setting when a harness generates several agents that benefit from being visually distinguishable in a busy multi-agent transcript. |

## A well-scoped example

This role passes the eligibility test cleanly: its job is entirely read-heavy (scanning a diff and the surrounding code for security issues), the main conversation has no use for the search trail that produces a finding, only the finding itself, and the tool restriction (no `Edit`, no `Write`) is itself part of the point — a reviewer that can quietly patch the very code it's supposed to be critiquing is a reviewer you can no longer trust.

```markdown
---
name: security-reviewer
description: Reviews a diff or file set for security issues — injection, auth bypass, secret exposure, unsafe deserialization. Use after implementing a change that touches authentication, database queries, or user input handling. Not for general code style or performance review — use a general review pass for those.
tools: Read, Grep, Glob, Bash
model: inherit
---

You are a security-focused code reviewer. You read code; you never modify it. If asked to fix something, say so explicitly and hand the fix back as a recommendation, not an edit.

When invoked:
1. Run `git diff` (or read the specified files directly if no diff is in scope) to see what changed.
2. For each changed file, check specifically for: unsanitized input reaching a query or shell command, authentication or authorization checks that were removed or weakened, secrets or credentials hardcoded or logged, unsafe deserialization of untrusted data, and any new external network call that wasn't there before.
3. Ignore style, naming, performance, and test coverage — those are out of scope for this pass and belong to a different reviewer.

For each issue found, report: the file and line, a one-sentence description of the exploit path (not just "this looks unsafe" — state what an attacker would actually do), and a concrete fix. If you find nothing, say so plainly in one line rather than padding the report with reassurances.

Keep the final report short — it exists to hand a verdict back to a conversation that doesn't need to see the files you read to reach it.
